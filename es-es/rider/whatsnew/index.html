Novedades de Rider

Rider 2019.2 incorpora nuevas sorpresas al depurador y mejora la compatibilidad con el desarrollo Xamarin iOS. La IU es ahora más clara, más cómoda y más coherente. Esta versión también trae Code Vision a F#, cobertura de código para Mono, y finalmente es compatible con controles de terceros y de 32 bits en el diseñador Windows Forms.

Mejoras del depurador

La pestaña “Variables” cuenta con una nueva funcionalidad denominada ‘Fijar en la parte superior’. Utilícela para marcar con una estrella las propiedades o los campos de un objeto que más le interesen, para mantenerlos en la parte superior de la lista. 

Las acciones ‘Instrucción Ejecutar hasta’ e ‘Instrucción Saltar a’ ahora cuentan con iconos tras el cursor en el editor.

También hemos añadido numerosas mejoras de rendimiento para el cálculo de pilas y avance.

Para obtener más información, consulte Novedades del depurador para Rider 2019.2: Fijar en la parte superior, Acciones flotantes y Marcos de pilas rediseñados.

Compatibilidad con Xamarin

Esta versión añade un par de funcionalidades útiles que le gustarán en el desarrollo de Xamarin iOS.

Para empezar, puede publicar un archivo .ipa (iOS App Store Package) para un proyecto de Xamarin iOS a un dispositivo iOS directamente desde el IDE. Seleccione un proyecto de iOS en Solution Explorer View, abra el menú contextual y haga clic en el elemento “Archivar para publicar”.

En segundo lugar, puede gestionar perfiles de aprovisionamiento desde la ventana Propiedades (Configuraciones | Firma de paquetes) para un proyecto de Xamarin iOS.

Diseñador Windows Forms

Rider 2019.2 incorpora compatibilidad con controles de terceros desde Telerik, DevExpress, Syncfusion, ComponentOne e Infragistics. Puede cargar un conjunto de controles en Designer Toolbox y, a continuación, utilizarlos en los formularios de aplicación y controles de usuario. Tenga en cuenta que algunos controles de DevExpress que hacen uso intensivo de la API de Visual Studio no funcionarán. Actualmente estamos buscando un paliativo.

Aparte de eso, hemos añadido compatibilidad con controles de 32 bits, de modo que ahora puede utilizar sus componentes de 32 bits en el diseñador.

Para obtener más información, consulte Uso de controles de Windows Forms de terceros con Rider.

Cobertura de código de aplicaciones Mono y pruebas de unidades

Además del análisis de cobertura de aplicaciones y pruebas .NET Core hemos añadido compatibilidad con el Marco de trabajo Mono. Ya puede cubrir tus pruebas de unidades de .NET Framework utilizando tiempo de ejecución Mono en Linux, macOS y Windows.

IL Viewer

¡Hemos traído a Rider la ventana de herramientas IL Viewer de ReSharper! IL Viewer muestra instrucciones IL para la clase o método que tenga abierto en ese momento en el editor. El editor e IL Viewer se sincronizan: al situar el signo de intercalación en un símbolo de su código se resalta el IL correspondiente, y viceversa. Al pasar el cursor por una instrucción IL, Rider muestra una descripción emergente con su descripción, obtenida de MSDN y la norma ECMA-335 (Partition III: CIL Instruction Set).

Mejoras de rendimiento

Tras publicar Rider 2018.3, hemos recibido muchos comentarios sobre su rendimiento, por lo que nos hemos centrado en mejorar esta área:

  • Desde que hemos movido la asistencia de Xamarin Android a un complemento separado no empaquetado, se ha reducido considerablemente el tiempo transcurrido entre que se hace clic en el icono de Rider y aparece la pantalla de bienvenida con la barra de progreso. Otro aspecto positivo es que el instalador ahora ocupa 150 MB menos.
  • En una de nuestras publicaciones anteriores, no se podía ejecutar NGen (Generador de imágenes nativas) para los binarios de Rider. Como resultado, abrir una solución en Rider tardaba 8 segundos más en Windows porque el proceso Compilador JIT tenía que ejecutarse en tiempo de ejecución. Ahora hemos vuelto a ejecutar NGen y hemos ganado 8 segundos en cada solución abierta.
  • Hemos hecho un par de optimizaciones en el arranque: hemos dejado de utilizar el motor ReSharper en el soporte JavaScript/TypeScript y hemos cambiado la carga del modelo de proyecto para que sea asíncrona. Como resultado, los arranques en caliente ahora son significativamente más rápidos (¡hasta el doble!).
  • Hemos implementado una carga de backend anticipada para acelerar 3 segundos la carga del servicio de lenguaje de ReSharper. Notará la diferencia si la configuración “Reopen last solution on startup” (Volver a abrir la última solución al iniciar) está activada o si actúa con gran rapidez y hace clic en una solución en la pantalla de bienvenida en menos de un segundo.

Compatibilidad con Razor

Todo el contenido sobre los lenguajes relacionados con la web (como JavaScript o CSS) en las páginas de Razor se ha movido del motor ReSharper al motor WebStorm. Esto significa que obtendrá mejor compatibilidad con marcos de trabajo y tecnologías de frontend de última tecnología: disfrute de la compatibilidad con Emmet y asistencia avanzada a la codificación para Angular, React, Vue.js y Meteor directamente en sus vistas Razor.

Dado que las vistas Razor eran el último lugar en el que se empleaba la compatibilidad web con ReSharper, ahora solo el motor WebStorm analiza su JavaScript y CSS. Puesto que los ensambles relacionados con ReSharper ya no se cargan al iniciar Rider, se han reducido los tiempos de carga de soluciones y de inicio de Rider.
