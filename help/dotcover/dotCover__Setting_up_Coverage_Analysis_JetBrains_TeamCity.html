在JetBrains TeamCity上设置承保范围

通常，.NET解决方案的持续集成（CI）构建由一系列任务组成，这些任务涉及测试的编译和运行。就JetBrains TeamCity CI服务器而言，这些任务称为构建步骤。随着TeamCity与dotCover Console Runner捆绑在一起，您只需启用相应的构建步骤选项即可收集代码覆盖率数据。TeamCity收集的coverage结果可以在TeamCity中查看为报告，也可以下载为coverage快照并在Visual Studio中查看。

例如，我们有一个简单的解决方案，它由两个项目组成：一个代表应用程序，另一个代表测试程序集。我们的任务是建立持续的集成构建，以编译应用程序，运行测试并收集测试覆盖率统计信息。请注意，工作流程因您使用的单元测试运行程序而异：

  • MSTest / NUnit的

  • 的xUnit

  • MSpec 。

MSTest / NUnit

TeamCity开箱即用提供MSTest和NUnit测试运行程序，因此，通常，您所需要做的就是添加具有相应运行程序的构建步骤。

在TeamCity中设置MSTest / NUnit测试的覆盖率分析

  1. 在TeamCity中，创建一个新的生成项目并设置VCS根目录。然后，我们添加一个新的构建步骤：创建一个新项目，并在项目内部创建一个新的构建配置 。

  2. 添加第一个构建步骤，将构建应用程序。在我们的特定示例中，我们使用Visual Studio（sln）运行器 （要求将Visual Studio安装在构建代理上），但是您还有许多其他构建方式的选择：直接运行MSBuild，运行批处理文件等。

    请注意，我们在这里所做的只是构建解决方案。尚无覆盖设置。

  3. 添加一个运行测试的构建步骤。如果使用MSTest测试，请在Runner type中选择Visual Studio测试 。我们的示例解决方案使用NUnit测试，因此，我们选择NUnit 。在这里，我们还指定NUnit版本，.NET运行时参数以及测试程序集的路径。

    如果使用Visual Studio测试，则参数看起来类似：您指定MSTest / VSTest版本和测试文件名。

  4. 现在，是时候建立覆盖范围了。在.NET Coverage工具中 ，选择JetBrains dotCover 。

  5. 在“程序集过滤器”中 ，添加要覆盖的程序集（仅是程序集的名称），并在其前面加上+:并筛选出您不希望覆盖的内容–: 。在我们的示例中，我们排除了MainTests包含正在运行的测试的程序集。

    此外，您可以使用属性过滤器排除标有特定属性（例如， ObsoleteAttribute ）从覆盖范围统计信息中。

  6. 保存配置并运行以查看其工作原理。

  7. 构建完成后，“ 概述”选项卡将包含有关代码覆盖率的简短报告。

    要查看详细信息，请切换至“ 代码覆盖率”选项卡。在这里，您可以查看有关测试涵盖了多少类/方法的详细统计信息。甚至深入研究各个类并检查代码覆盖率。现在，构建工件包含压缩的Coverage文件。

  8. 请注意，在实际环境中，您可能有两种不同的构建配置：一种构建二进制文件，另一种收集代码覆盖率。在这种情况下，您应该使用快照依赖性，以便第二个构建配置可以找到源文件。否则，您将看到Source code is not available尝试查看源代码时出现错误消息。

    要解决此问题，请按照TeamCity文档中的说明进行操作。

单位

运行和获取xUnit测试的主要问题是，TeamCity不支持xUnit。这会产生两个必须解决的问题：

  • 必须提供xUnit软件包才能构建解决方案。
    将二进制库提交给VCS并不是一个好习惯，因此，还原软件包必须是编译阶段之前的一个单独构建步骤。

  • 必须提供xUnit测试运行程序并手动运行。
    好消息：

    • 运行程序以NuGet软件包的形式分发，因此可以在解决方案中直接引用它，然后在构建过程中将其还原（请参阅a）。

    • 由于跑步者以TeamCity服务消息的格式提供结果，因此无需采取其他任何操作即可获得测试结果。

在TeamCity中设置xUnit测试的覆盖率分析

  1. 在Visual Studio中，打开您的解决方案。在测试项目中，添加对xunit.runner.console NuGet软件包。

  2. 在TeamCity中，创建一个新的生成项目并设置VCS根目录。然后添加一个新的构建步骤：创建一个新项目，并在项目内部创建一个新的构建配置 。

  3. 确保，TeamCity的配置为的NuGet服务器：在管理|工具会找到NuGet.exe表，并检查其中是否包含所需的NuGet版本。如果没有这样的表，请通过单击“ 安装工具...”按钮并按照所需的步骤将NuGet添加到TeamCity。

  4. 添加一个构建步骤，该步骤还原该解决方案引用的NuGet软件包。使用NuGet Installer作为Runner类型 。

  5. 添加一个构建步骤，将构建应用程序。在我们的特定示例中，我们使用Visual Studio（sln）运行器 （要求将Visual Studio安装在构建代理上），但是您还有许多其他构建方式的选择：直接运行MSBuild，运行批处理文件等。

    请注意，我们在这里所做的只是构建解决方案。尚无覆盖设置。

  6. 添加一个运行测试的构建步骤：

    • 由于xUnit运行程序将被简单地执行.exe文件，在Runner类型中选择.NET Process 。

    • 由于运行程序是作为解决方案中的软件包提供的，因此您可以指定其相对于解决方案工作目录的路径，例如
      %system.teamcity.build.workingDir%SimpleTestsProject\packages\xunit.runner.console.2.3.1\tools\net452\xunit.console.exe

    • 命令行参数字段必须包含已编译测试程序集的路径。

  7. 现在，是时候建立覆盖范围了。在.NET Coverage工具中 ，选择JetBrains dotCover 。

  8. 在“程序集过滤器”中 ，添加要覆盖的程序集（仅是程序集的名称），并在其前面加上+:并筛选出您不希望覆盖的内容–: 。在我们的示例中，我们排除了MainTests包含正在运行的测试以及与xUnit程序集相关的所有内容的程序集。

    此外，您可以使用属性过滤器排除标有特定属性（例如， ObsoleteAttribute ）从覆盖范围统计信息中。

  9. 保存配置并运行以查看其工作原理。

  10. 构建完成后，“ 概述”选项卡将包含有关测试结果和代码覆盖率的简短报告。

    要查看详细信息，请切换至“ 代码覆盖率”选项卡。在这里，您可以查看有关测试涵盖了多少类/方法的详细统计信息。甚至深入研究各个类并检查代码覆盖率。现在，构建工件包含压缩的Coverage文件。

规格书

MSpec由TeamCity开箱即用地提供部分支持：它具有相应的构建步骤运行器，但是MSpec测试运行器不在捆绑包中，必须手动安装到TeamCity代理。幸运的是，运行程序以NuGet软件包的形式分发，因此可以在解决方案中直接引用它，然后在构建过程中将其还原。

在TeamCity中设置MSpec测试的覆盖率分析

  2. 在TeamCity中，创建一个新的生成项目并设置VCS根目录。然后添加一个新的构建步骤：创建一个新项目，并在项目内部创建一个新的构建配置 。

  4. 添加一个构建步骤，该步骤还原该解决方案引用的NuGet软件包。使用NuGet Installer作为Runner类型 。

  5. 添加一个构建步骤，将构建应用程序。在我们的特定示例中，我们使用Visual Studio（sln）运行器 （要求将Visual Studio安装在构建代理上），但是您还有许多其他构建方式的选择：直接运行MSBuild，运行批处理文件等。

    请注意，我们在这里所做的只是构建解决方案。尚无覆盖设置。

  6. 添加一个运行测试的构建步骤：

    • 在Runner类型中选择MSpec 。

    • 由于运行程序是作为解决方案中的软件包提供的，因此您可以指定其相对于解决方案工作目录的路径，例如
      %system.teamcity.build.workingDir%SimpleTestsProject\packages\Machine.Specifications.Runner.Console.0.9.3\tools\mspec-clr4.exe
      确保使用与项目目标.NET Framework版本相对应的运行器版本。

    • 在“从以下位置运行测试”中 ，指定已编译测试程序集的路径。

    • 如果需要，请在“ 包括规格”和“ 排除规格”中指定必须包括/ 排除的规格 。

    此外，您可以使用属性过滤器排除标有特定属性（例如， ObsoleteAttribute ）从覆盖范围统计信息中。
