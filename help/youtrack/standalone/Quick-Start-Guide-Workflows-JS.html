JavaScript Workflow快速入门指南

本《快速入门指南》介绍了开始使用JavaScript工作流所需的基本概念。

  • 如果您想跳过热身，可以直接进入API 。

  • 如果你从来没有在YouTrack与工作流程的工作，开始与工作流程的更多背景信息。

权限

在开始之前，请确保您具有创建和编辑工作流程的权限。

  • 拥有“ 读取项目”和“ 更新项目”权限的用户可以创建自己的工作流程并将其附加到他们的项目中。

  • 具有“ 低级管理”权限的用户可以创建工作流程并将其附加到任何项目。

用于创建工作流程的控件仅在“ 工作流程”页面上可用，该页面可在“ 管理”菜单的“ 项目相关的设置”部分中访问。

创建工作流程

工作流本身只是一个容器。您可以使用工作流来收集要在项目中应用的一组规则。

您可以构建工作流以强制执行特定的业务逻辑，因此，在开始之前最好坐下来规划用例。

对于本指南，请让您自己成为项目经理。您刚刚在项目中添加了要用于对问题进行分组的字段，以便可以将进度报告给老板。您创建了一个名为Phase的字段，并添加了Development ， Testing和Ready值 。基本上，您的老板只需要查看在开发或测试中仍然存在多少问题，即可根据路线图衡量进度。

现在，您需要创建一个工作流，该工作流将根据应用于项目中问题的其他更改来设置此字段的值。您想自动设置该字段的值，因此您的项目团队不必担心自己设置值。

要创建工作流程：

  1. 在管理菜单的核心功能部分中，选择工作流程 。

  2. 单击创建工作流程按钮。

    • 将打开“ 新建工作流”对话框。

  3. 在“ 新建工作流”对话框中，输入名称和可选标题。工作流程名称遵循npm软件包的标准命名约定。有关更多信息，请参见https://docs.npmjs.com/files/package.json#name 。

    对于本指南，请使用阶段管理作为名称，使用阶段管理作为标题。

  4. 单击保存按钮。

    • 您的新工作流程将添加到列表中。

将模块添加到您的工作流程

这样就足够简单了，但是您才刚刚开始。现在，您有了一个可以存储脚本的容器。您可以在模块中将脚本添加到工作流中，这些模块可以包含不同类型的规则或自定义脚本。

您要在该项目中自动化的第一步是，当团队决定推迟任务并将其移至产品待办事项列表时，清除“ 阶段”字段的值。当团队推迟任务时，他们将“ 修订版本”字段的值更改为Backlog 。

将更改应用于问题时，将执行此操作，因此您要向工作流程中添加包含更改规则的模块。

要将模块添加到工作流程中：

  1. 在列表中找到新的工作流程。

  2. 单击工作流程名称以访问工作流程编辑器。

    • 工作流程将在工作流程编辑器中打开。

    • “ 新模块”面板在编辑器窗口中打开。

    • 现在，您可以使用模板将规则添加到工作流程中。

  3. 在“ 新建模块”面板中，单击“ 更改”按钮。

    • 将打开“ 新模块”对话框。

  4. 输入模块的名称，例如clear-phase-when-postponed ，然后单击保存按钮。

    • 新模块已添加到工作流程中。

    • 模块名称显示为链接。

    • 更改规则的模板会在编辑器中加载。

写下你的第一条规则

现在您可以编写第一个简单规则了。最好的进行方法是从一个开始TODO在模板中添加注释到下一个注释，逐行更新代码并删除所有不必要的注释。

编写第一个规则：

  1. 从模板中的第一个注释块开始：
    / ** *这是更改规则的模板。此规则定义将*应用于问题时发生的*。* *有关详细信息，请阅读《快速入门指南》：* https://www.jetbrains.com/help/youtrack/standalone/2017.4/Quick-Start-Guide-Workflows-JS.html * /

    除了向您展示如何向模块中添加注释块之外，该注释还告诉您该模板中定义了哪种类型的规则，并为您提供了该快速入门指南。由于您已经在这里，并且知道这种规则的作用，因此继续删除整个块。

  2. 查看第一行代码：
    var实体= require（'@ jetbrains / youtrack-scripting-api / entities'）;

    此变量声明引用了entities工作流程API中的模块。这意味着包含在entities可以按此规则访问模块。

    例如，如果存在属于其他模块的属性和方法，则workflow模块，只需添加一个require语句并指向您要引用的模块，如下所示：

    var工作流程= require（'@ jetbrains / youtrack-scripting-api / workflow'）;

    这里最重要的一点是，您需要熟悉工作流API，并且知道哪些模块包含工作流所需的属性和方法。对于此规则，您只想更新自定义字段中的值。可以从以下位置访问问题及其自定义字段entities模块，因此您可以保留require模板中内置的语句。

  3. 设置exports.rule属性。

    脚本可以将一个规则恰好导出到此属性。在模板中，使用entities.Issue.onChange方法。此方法声明了将更改应用于问题时触发的规则，因此您已全部准备就绪。您已经确定该脚本在加载模板时将作为更改规则处理。

  4. 设置一个值title属性。

    对于更改规则，此属性是可选的。您已经可以看到用于模块名称的值已复制到标题。您希望标题对用户友好，并帮助其他用户识别此规则的作用。对于此示例，将标题更改为Clear Phase when Fix version is set to Backlog 。

    设置标题后，删除TODO在它前面的那一行。

  5. 写一个guard条件

    这是您实际开始编写代码的地方。随着guard属性，您可以定义要应用规则必须满足的条件。

    模板已将Guard声明为接受上下文作为参数的函数：

    卫队：function（ctx）{

    这仅表示为该代码定义的所有代码guard属性被视为函数。对于接下来要执行的动作，此函数必须返回一个值，其值为true 。

    您希望此规则仅在用户将“ 修订版本”字段的值更改为Backlog时才应用操作。以下护卫返回true当此更改应用于问题时：

    返回ctx.issue.fields.becomes（ctx。FixVersion，ctx。FixVersion。积压）;

    您可以将此警戒贴在TODO在警卫财产内。

  6. 设置一个action 。

    随着action属性，您定义满足保护条件时要应用的更改。

    您可以看到模板已经将动作声明为接受上下文作为参数的函数：

    动作：function（ctx）{

    然后声明一个局部变量issue并设置值以引用上下文中的问题：

    var issue = ctx.issue;

    您将需要为复杂的规则使用变量声明，以更新多个问题属性，但是对于一个简单的规则，则不需要它。您需要做的就是指定在满足保护条件时要应用的操作，这将清除“ 阶段”字段中的值：

    ctx.issue.fields。相位=空;

    您可以复制此代码块并将其粘贴到不必要的变量声明之上，然后删除下一个TODO 。

    您会注意到，模板使用右括号将函数的主体关闭，并使用逗号将该块与下一个属性分开（ }, ）。

  7. 加requirements 。

    的requirements属性列出了该规则正常运行所需的所有组件。这包括自定义字段，问题链接类型，项目，保存的搜索等的定义。要了解有关需求的更多信息，请参阅需求 。

    对于您的规则，您需要定义两个自定义字段的要求： Fix version和Phase 。

    首先，定义“ 修订版本”字段将数据存储为ProjectVersion类型，然后指定该字段必须包含Backlog在一组值中。

    FixVersion：{类型：实体。ProjectVersion.fieldType，名称：“修复版本”，积压：{}}，

    首先，您会注意到FixVersion不是自定义字段的实际名称。这是一个别名。对于包含空格和其他非字母字符的自定义名称，请使用别名，因为它使您不必在引号和方括号中设置这些值。自定义字段的实际名称在name属性。的最终属性FixVersion指定自定义字段必须包含值Backlog 。

    然后，为“ 相位”字段设置要求。

    阶段：{类型：实体。EnumField.fieldType，}

    在这里，您需要做的就是要求该字段存在并存储一个EnumField 。您无需为此字段输入任何值，因为规则会将值设置为null 。

    复制这些代码块并将其粘贴到最后一个代码的顶部TODO评论。

  8. 单击保存按钮以存储您的更改。

  9. 在工作流编辑器中打开“ 更多操作”菜单，然后选择“ 重新设置代码格式” 。完成的工作流程应如下所示：

然后写另一个！

那很容易。还有另一件事。只要记住您从第一条规则中学到的知识，然后向工作流程中添加第二个模块即可。

您要自动化的下一个动作是在对“ 状态”字段应用更改时设置“ 相位”字段的值。当对问题应用更改时，也会执行此操作，因此您要添加另一个更改规则。

    现在，写下更改“ 相位”字段值的条件。将条件和动作定义为一系列“如果”：
