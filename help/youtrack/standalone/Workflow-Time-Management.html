时间管理

该工作流根据问题在特定状态下花费的时间来定义状态转换和通知。

名称

@ jetbrains / youtrack-workflow-time-management

自动连接

没有

模组

使用预定义的时间约束 （状态机规则） 为“状态”字段定义转换

用例

该工作流程最初是从提交的请求（ JT-6331 ）中获取的。

提交此问题的用户希望跟踪客户，潜在客户和团队成员报告的问题，并确保在特定时间内解决问题。

当客户报告问题时：

  • 支持工程师有一个小时的时间来确认该问题。

  • 然后，支持工程师有四个小时的时间来重现该案例或进行讨论以重现该案例。

    • 如果一天之内未重现问题，则将与支持经理联系，并通知销售部门。

    • 三天后，销售部门组织一次客户拜访，以现场核实问题。

  • 如果将问题归类为错误，则将问题分配给研发部门。

模组

该模块包含一个状态机规则，该规则定义项目中问题的状态转换，并确定问题在每种状态下可以花费的时间。

使用预定义的时间限制为“状态”字段定义转换

  1. 初始状态为Submitted 。

  2. 当问题进入“已提交”状态时，“ 子系统”字段将设置为“ 支持” 。支持工程师被设置为受让人 。

  3. 从已提交状态，问题可以转换为以下状态之一：

    • 如果一小时后状态未更改，则状态将自动设置为Overdue 。

    • 在事件（动作） 再现时 ，状态设置为Open 。

    • 在事件（动作）未完成时 ，状态设置为未完成 。

  4. 当状态设置为Overdue时 ，将通知发送给受让人 （如果分配了问题），子系统所有者（如果定义了Support子系统的所有者）或项目所有者（如果先前的条件都不成立） ）。

  5. 从“ 过期”状态，问题可以转换为以下状态之一：

    • 再现事件（动作）时，状态设置为Open 。

    • 在事件（动作）未完成时 ，状态设置为未完成 。

  6. 从打开状态，问题可以转换为以下状态之一：

    • 如果四个小时后状态未更改，则状态将自动设置为等待再现 。

    • 在事件（操作） 已批准时 ，状态将设置为已批准 。

    • 在事件（动作）未完成时 ，状态设置为未完成 。

    • 在事件（动作） 无法重现时 ，状态设置为无法重现 。

  7. 从等待重现状态，问题可以转换为以下状态之一：

    • 如果一天后状态保持不变，则将通知发送到支持子系统（如果已定义）的所有者或项目所有者。

    • 如果三天后状态保持不变，则会向支持子系统（如果已定义）的所有者或项目所有者发送单独的通知。

    • 在事件（操作） 已批准时 ，状态将设置为已批准 。

    • 在事件（动作）未完成时 ，状态设置为未完成 。

    • 在事件（动作） 无法重现时 ，状态设置为无法重现 。

  8. 当状态设置为批准时 ，规则将验证是否也设置了受让人 。

  9. 从已批准状态，问题可以过渡到以下状态之一：

    • 事件（动作） 固定的状态下被设置为固定状态。

    • 如果事件（操作） 已过时，则将状态设置为（应过渡）为“ 过时”状态。

  10. 从固定状态，问题可以过渡到以下状态之一：

    • 在事件（操作） verify时 ，状态设置为“ 已验证”状态。

    • 重新打开事件（操作）时，状态将设置为“ 打开”状态。

  11. 从“无法复制” ，“ 未完成” ，“ 过时”和“ 已验证”状态，问题可以在操作重新 打开后转换为“ 打开”状态。
