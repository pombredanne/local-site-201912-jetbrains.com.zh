提交并推送更改

将新文件添加到Git存储库中或修改了已经在Git版本控制下的文件，并对它们的当前状态感到满意之后，就可以共享工作结果。这包括在本地提交它们，以将存储库的快照记录到项目历史记录中，然后将它们推送到远程存储库，以便其他人可以使用它们。

本地提交更改

  1. 在页面中选择要提交的文件或整个更改列表。 的“本地更改”标签 版本控制窗口Alt+9然后按Ctrl+K或点击在工具栏上。

    将打开“ 提交更改”对话框，其中列出自上次提交以来已修改的所有文件以及所有新添加的未版本控制的文件。

    您可以使用“ 提交更改”对话框或直接在“ 本地更改”视图中管理提交。使用“ 从本地更改提交”，而不会在屏幕上显示“提交”对话框选项 版本控制| RubyMine设置的“ 提交对话框”页面Ctrl+Alt+S选择首选的行为。

  2. 查看要提交的文件列表，清除不想包括在提交中的文件和目录旁边的复选框。排除文件中的更改将保留在活动更改列表中，您可以稍后提交。

  3. 输入提交消息。

    您可以点击从最近的提交消息列表中选择。

    您可以在推送提交之前编辑提交消息 。

    自定义提交消息规则在里面 设置/首选项对话框Ctrl+Alt+S选择版本控制|提交对话框 。还有一个快速修复程序和“ 重新格式化”操作，用于包装长行或重新格式化消息。

  4. 如有必要，在将所选文件提交到本地存储库之前 ，选择要让RubyMine执行的“ 提交前”操作。

  5. 如有必要，请在Git部分中选择以下选项：

    • 作者 ：如果您要提交其他人所做的更改，则可以指定这些更改的作者。

    • 修改提交 ：将本地更改添加到最新提交（有关详细信息，请参见将分段的更改与上一次提交合并（修改提交） ）。

    • 退出提交 ： 选择是否要注销提交以证明您将要签入的更改已由您进行，或者由您对提交的代码负责。

      启用此选项后，将在提交消息的末尾自动添加以下行： 签名人：

  6. 单击提交或将鼠标悬停在此按钮上以显示可用的提交选项：

    • 提交并推送 ：提交后立即将更改推送到远程存储库。

      您也可以按Ctrl+Alt+K从“ 提交”对话框中调用“ 提交和推送”操作。

    • 创建补丁 ：根据您将要提交的更改生成补丁 。在打开的“ 创建补丁”对话框中，键入补丁文件的名称，并指定是否需要反向补丁。

    • 远程运行 ： 运行您的个人构建 。仅当您登录TeamCity时，此选项才可用。有关详细信息，请参阅TeamCity插件文档 。

提交文件的一部分

有时，当您进行与特定任务相关的更改时，您还会应用影响同一文件的其他不相关的代码修改。将所有此类更改都包含在一个提交中可能不是一个好选择，因为要检查， 还原 ， 选择它们等等会更加困难。

RubyMine使您可以通过以下方式之一分别提交此类更改：

  • 选择要包含在“ 提交更改”对话框中的提交权限中的修改后的代码块 ，并保留其他更改，以便以后可以提交。

  • 在编辑代码时， 将不同的代码块动态地放入不同的变更列表中 ，然后分别提交这些变更列表。

选择要提交的块

  1. 调用 “ 提交更改”对话框。

  2. 点击DIFF显示库版本和所选文件的本地版本之间的差异。

  3. 选中要提交的已修改或新添加的代码的每个块旁边的复选框，并取消选择其他更改：

  4. 点击提交 。未选择的更改将保留在当前更改列表中，以便您可以分别提交它们。

将更改放入差异更改列表

  1. 在编辑器中对文件进行更改时，请在装订线中单击相应的更改标记 。

  2. 在出现的工具栏中，为修改后的代码块选择目标更改列表（或创建一个新的更改列表）：

  3. 分别提交每个变更列表。

您还可以创建一个新的变更列表并将其激活 ，然后将其后所做的所有更改都包含在该变更列表中，而在此之前所做的任何修改都将保留在原位置。

将更改推送到远程存储库

在进行更改之前，请与远程同步，并确保您的存储库本地副本是最新的，以避免冲突。

RubyMine允许您将更改从当前分支上传到其跟踪分支或任何其他远程分支。

  1. 按Ctrl+Shift+K或选择VCS | Git |从主菜单中按 。将打开“ 推送提交”对话框，显示所有Git存储库（用于多存储库项目）， 并列出自上次推送以来在每个存储库中当前分支中进行的所有提交。

    如果您的项目使用的多个存储库没有被同步控制，则默认情况下仅选择当前存储库（有关如何启用同步存储库控制的详细信息，请参阅版本控制设置：Git ）。

  2. 如果存储库中没有远程服务器，则将显示“ 定义远程”链接。单击此链接，然后在打开的对话框中指定远程名称和URL。它会被保存，您以后可以通过VCS |编辑它。 Git |遥控器 （有关详细信息，请参阅添加远程存储库 ）。

  3. 如果要修改要推送到的目标分支，可以单击分支名称。标签变成一个文本字段，您可以在其中键入现有的分支名称或创建新的分支。您也可以单击右下角的“ 编辑所有目标”链接以同时编辑所有分支名称。

    请注意，您不能更改本地分支：每个选定存储库的当前分支将被推送。

  4. 如果要在推送更改之前预览它们，请选择所需的提交。右侧窗格显示了所选提交中包含的更改。您可以使用工具栏按钮来检查提交详细信息。

  5. 准备就绪后，单击“ 推”按钮，然后从下拉菜单中选择要执行的操作：“ 推”或“ 强制推” 。

    仅当“ 受保护的分支”字段中未列出当前分支时，这些选择选项才可用（请参阅“ 版本控制设置：Git” ），否则，您只能执行push操作。

如果推送被拒绝，则更新您的工作副本

如果推被拒绝，因为你的工作拷贝是过时的，RubyMine的显示推送拒绝对话，提供了自动更新，如果当前分支的推动被拒绝的选项Git的设置未选择对话框中设置 首选项的页面。请执行下列操作：

  1. 如果您的项目使用多个Git存储库，请指定要更新的存储库。如果要更新所有存储库，则无论是否拒绝推送 ，都选择“ 更新不拒绝存储库”选项。如果清除此选项，则仅更新受影响的存储库。

  2. 如果希望RubyMine在下一次使用在此对话框中选择的更新方法拒绝推送时，以静默方式应用更新过程，请选择“ 记住更新方法”选项，并在将来进行静默更新选项。

    离开此对话框后，将选中“ 设置 首选项”对话框的“ Git设置”页面中的“ 如果拒绝当前分支的推送 ，则自动更新”复选框，并且应用的更新方法将成为默认方法。

    要更改更新策略，请取消选择此选项，以在下次拒绝当前分支的推送时调用“ 拒绝推送”对话框，应用其他更新过程，然后再次选择“ 记住更新方法”选择选项。

  3. 通过分别单击Rebase或Merge按钮来选择更新方法（ rebase或merge ）。

我什么时候需要使用推力？

当您运行push时 ，如果远程存储库中有丢失的更改并且您打算用存储库的本地副本覆盖，则Git将拒绝完成操作。通常，在通过更改进行更新之前，需要执行拉动以与远程同步。

的--force push命令禁用此检查，并允许您覆盖远程存储库，从而擦除其历史记录并导致数据丢失。

您可能仍需要执行的可能情况--force push是当您对推送分支进行重新设置基准然后想将其推送到远程服务器时。在这种情况下，当您尝试推送时，Git将拒绝您的更改，因为远程引用不是本地引用的祖先。如果在这种情况下执行拉动 ，最终将得到分支的两个副本，然后需要合并它们。

如果您决定强行推送重新分支的分支并且您正在团队中工作，请确保：

  • 没有人拉过您的分支并对它进行一些本地更改

  • 所有未完成的更改均已提交并推送

  • 您拥有该分支机构的最新更改
