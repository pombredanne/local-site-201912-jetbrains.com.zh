TeamCity 2019.1帮助

清洁结帐

Clean Checkout （也称为“ Clean Sources”）是一项操作，可确保下一个版本将获得从VCS整个获取的源的副本。将删除Build Checkout目录的所有内容，并从版本控件中重新获取源。

强制执行清洁结帐

如果签出目录的内容是由外部过程通过添加新文件，修改或删除现有文件来修改的，则建议执行干净签出。

您可以从“构建配置主页”（右上角的“ 操作”下拉列表）对构建配置强制执行“清洁源”操作，也可以使用“ 杂项”部分的“ 代理详细信息”页面对“ 代理上的清洁源”选项对代理强制执行“清洁源”操作。该操作将打开要清除其源的代理程序/内部配置列表。

如果将特定文件夹设置为Build Checkout目录（而不是使用默认文件夹），则应记住该目录的所有内容将在干净签出过程中删除。

TeamCity为源维护内部缓存，以优化与VCS服务器的通信。缓存在清理期间重置。要解决源更新问题，可能需要使用诊断|手动重置缓存。 Web UI中的“ 缓存”标签或通过删除< TeamCity Data Directory > /system/caches目录。

自动清理结帐

如果选中“ 创建/编辑构建配置 > 版本控制设置”页面上的“ 在构建之前清除所有文件 ”选项，则还可以启用在每次构建之前自动清除源。如果选中此选项，TeamCity将在每次构建之前执行完整的检出。如果未启用干净签出，TeamCity会将签出目录中的源增量更新为所需状态。

TeamCity尝试检测检出目录中的源是否与预期状态不对应，并在这种情况下触发干净检出以确保源适当。这意味着在某些情况下，即使未在VCS设置中启用并且用户未从Web UI要求，TeamCity仍可以检测到有必要进行彻底结帐。在这种情况下，将删除checkout目录的所有内容，并由源从头开始重新填充它。如果该决策有任何详细信息可用，它们将在与结帐相关的日志记录之前添加到构建日志中。

以下是TeamCity执行自动清理结帐的情况摘要：

  • 如果使用构建配置的“版本控制设置”中的“ 在构建前先清除checkout目录中的所有文件 ”选项启用了此功能

  • 找不到构建签出目录或该目录为空（第一次在代理上启动构建配置，或者自上次构建以来该目录已消失）。还包括以下内容：

    • 在配置的（或默认的）时间内，没有在特定的检出目录中运行任何版本，并且该目录变为空。在自动结帐目录清理中查看更多信息

    • 在较早的版本之一中，磁盘上的可用空间不足，并且目录已删除

  • 用户从Web UI调用了“强制执行干净签出”操作以进行构建配置或代理

  • 可以通过“自定义运行构建”对话框（选择“在构建之前清除结帐目录中的所有文件”选项）或带有相应选项的触发器来触发构建

  • 该生成是由启用了“在生成之前清除checkout目录中的所有文件”选项的Schedule触发器触发的，或者是在构建链的一部分中触发的，其中最高的构建是通过schedule触发器中的设置触发的，而“适用于所有快照依赖项”也被选中

  • 构建配置的VCS设置已更改

  • 此目录中的先前构建是具有不同VCS设置的构建配置（仅当为具有单独VCS设置和VCS根的多个构建配置指定了相同的检出目录时，才可能发生）

  • 此目录中的上一个版本是基于最新修订版本构建的，而当前版本是（仅对于历史版本而言 ）

  • 在先前的构建过程中应用或回滚补丁时出现严重错误，因此TeamCity无法确保checkout目录包含文件的已知版本

  • 使用相应的选项启用了Build Files Cleaner（Swabra） ，并且检测到有必要进行干净签出

  • 自定义结帐目录包含特定于代理的参数，例如`％teamcity.agent.work.dir％（8.1之前的版本）
