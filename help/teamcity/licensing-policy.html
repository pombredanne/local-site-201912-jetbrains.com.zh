版本

TeamCity有两个版本： 专业版和企业版 。
除了允许的最大构建配置数量外，所有功能的版本均相同。
两种版本使用相同的TeamCity发行和安装。您可以通过输入适当的许可证密钥来切换到企业版。切换版本时，所有数据都会保留。

每个TeamCity Web UI页面的页脚以及“ 管理” >“ 许可证”页面上以及服务器启动时的teamcity-server.log中都注明了正在使用的当前版本。

专业版不需要任何许可证密钥，可以免费使用。与企业版唯一的功能差异是最大数量的构建配置限制。限制为100个构建配置（在TeamCity 2017.2之前为20）。添加了每个代理程序许可证密钥后，它可以扩展10个。您可以使用专业版许可证安装多台服务器。

企业版需要许可证密钥，对构建配置数量没有限制，并且只要您使用最新的TeamCity版本 ，您就可以在许可证的维护期内从JetBrains获得TeamCity 支持 。

高可用性设置中的其他服务器使用来自主服务器的许可证，并且此时不需要单独的许可证。

每个TeamCity版本都捆绑有3个或更多的构建代理 。要使用比捆绑号码更多的代理，可以输入单独的构建代理许可证密钥。可以将其他代理添加到两个版本中。

除了专业版和企业版许可证外，还有另外两种许可证类型：

  • 评估 —具有有效期，并提供无限数量的代理和构建配置。要获取评估许可证，请使用TeamCity下载页面上的链接。每个主要TeamCity版本只能获得一次评估许可证。同一主要版本的TeamCity服务器不接受该站点的第二个评估许可证密钥。如果您需要扩展/重复评估，请联系我们的销售部门。
    每个TeamCity的EAP （预览版，不稳定）版本都附带60天的评估许可证。

  • 开源 -这是为开源项目授予的一种特殊类型的许可证，它是基于时间的，并且提供了无限数量的代理。请参阅页面上的详细信息

TeamCity许可策略对任何IDE插件或Windows托盘通知程序的实例数量没有施加任何限制。

构建配置数

企业版对构建配置的数量没有限制。

自TeamCity 2017.2起，专业版允许每个服务器进行100个构建配置 （早期版本中为20个）。从TeamCity 8.0开始，每个构建代理许可证密钥除了提供一个代理外，还为您提供了专业版中的10种构建配置。计算所有构建配置（即包括已归档项目中的配置）。

代理人数

TeamCity Professional版随附3个构建代理 。通过购买其他代理许可证密钥，可以添加更多的构建代理。

服务器许可证密钥可能包含更多的代理许可证，而不是默认的3。在jetbrains.com网站上声明的服务器许可证代理人数量记录了将可用的代理人总数。每个TeamCity版本（企业版和专业版）都可以使用单独的代理许可证密钥。有关购买代理许可证的更多信息，请参阅产品页面 。

代理商许可证的数量限制了TeamCity中可以授权的代理商的数量。许可证密钥没有绑定到特定的代理，它们只是限制了功能代理的最大数量。许可在本地（安装在TeamCity服务器计算机上）和远程代理之间没有区别。
当授权的代理数量超过可用的代理许可证时，服务器将停止启动任何内部版本，并在Web浏览器中向所有用户显示警告消息。

管理许可证

您可以在TeamCity Web UI的“ 管理” >“ 许可证”页面上输入新的许可证密钥并查看当前使用的密钥（包括许可证颁发日期和维护期限）。默认情况下，只有具有系统管理员角色的用户才能访问该页面。在页面上添加或删除许可证将立即应用。

单个许可证只能在任何给定时间在单个运行的TeamCity服务器上使用。除主要节点外，运行次要TeamCity节点目前不需要单独的许可证。

如果创建服务器的副本并同时运行两个服务器，则应确保每个许可证密钥仅在单个服务器上使用。您可以使用评估版（限时）许可证来运行服务器以进行测试/非生产目的。许可证未绑定到特定的服务器实例，计算机等。唯一的限制是许可证不能同时在多台服务器上使用。

如果您已经拥有许可证并购买了更多许可证，则可以请求 JetBrains销售将新许可证与已购买的许可证共同确定为新许可证，以使所有许可证具有相等的维护到期日期。然后按比例降低许可证的成本。
购买许多许可证时，欢迎您与我们的销售人员联系以获取批量折扣。

有效的TeamCity版本

对于所涵盖的TeamCity版本，TeamCity许可证是永久性的。这意味着您可以无限期地使用现有许可证运行涵盖的TeamCity版本，并且许可证对于该TeamCity版本仍然有效。
每个TeamCity许可证（包括Enterprise Server和Agent）都有一个维护期 （通常为1年）。许可证密钥对在购买许可证之前发布的任何版本的TeamCity以及在维护期内发布的任何版本均有效。对于主要版本/次要版本有效的许可证（前两个版本编号中的更改）也被视为对相应的错误修正更新（第三个版本编号中的更改）有效。

在升级到新的TeamCity版本之前，请使用新版本检查现有许可证的有效性。
如果某些许可证的维护期限未涵盖新的TeamCity服务器的有效发布日期 ，则相应的许可证在TeamCity版本中将无效，因此需要续订 。通常，如果新许可证的日期与要更新的许可证的维护终止日期相同，则许可证的更新费用约为新许可证价格的50％。

当有新版本可用时，TeamCity在Web UI中显示通知，并警告您任何许可证密钥与该新版本不兼容。服务器运行状况报告的“全局配置项目”中还将显示有关新TeamCity版本的通知，系统管理员可以看到。系统管理员可以使用“某些许可证不兼容”消息中的链接快速导航到“ 许可证”页面，其中所有不兼容的许可证都将带有警告图标。有关服务器上安装的许可证密钥的信息是安全的，因为它不是通过Internet发送的。

建议定期升级，因为新版本包含很多修复程序（当然还有新功能）。

请注意，TeamCity 电子邮件支持仅涵盖最新的TeamCity版本，并且只能提供给企业服务器许可证的维护期限未到期的客户。

许可到期

如果从服务器上删除了企业许可证密钥，或者评估许可证已过期，或者TeamCity服务器已升级到可用企业许可证的维护窗口之外发布的版本，TeamCity会自动切换到专业模式。

如果构建配置的数量或授权代理的数量超过有效许可证施加的限制，则服务器将停止启动任何构建（暂停构建队列），并在Web浏览器中向所有用户显示警告消息。

Build Agent许可证的工作方式与Server Licenses相同。如果将服务器升级到代理许可证维护窗口未涵盖的版本，则该代理许可证将过期。

一旦输入了足以覆盖服务器配置的有效许可证密钥，便会再次开始构建。

获得许可证的方式

存在以下几种将服务器切换到企业模式的方法：

  • 购买企业服务器许可证；

  • 在下载页面上申请60天的评估许可证（请参阅上面的详细信息）；

  • 使用TeamCity EAP版本 （不稳定，但捆绑了60天的非限制性许可证）；

  • 仅将TeamCity用于开放源代码项目，并申请开放源代码许可证 。

从以前的版本升级

从TeamCity 5.x和更高版本升级

每个许可证都有维护期限（自购买之日起通常为一年）。该许可适用于维护期内发布的任何TeamCity版本。升级之前，请检查许可证的维护期限。

从TeamCity 4.x升级到TeamCity 5.0及更高版本

需要升级TeamCity的早期版本的许可证，请参阅官方网站上“ 许可和升级”部分的详细信息。

从TeamCity 3.x升级到TeamCity 4.0

TeamCity 3.x Enterprise服务器许可证的所有者免费升级到TeamCity 4.x Enterprise Edition。TeamCity 3.x Build Agent许可证与TeamCity 4.0的专业版和企业版兼容。

从TeamCity 1.x-2.x升级到TeamCity 4.0

在2008年12月5日之前购买的任何TeamCity 1.x-2.x许可证都可以用作TeamCity 4.0专业版和企业版的一个TeamCity 4.0 Build Agent许可证。此外，TeamCity 1.x-2.x客户可以免费获得一份 TeamCity Enterprise Server许可证。要申请您的企业服务器许可证，请与销售部门联系以获取您的TeamCity 1.x-2.x许可证之一。

使用IntelliJ IDEA 6.0许可证密钥进行升级

在2006年7月12日到2007年1月15日之间购买的任何IntelliJ IDEA 6.0许可证都可以用作一个TeamCity 4.0 Build Agent许可证。此外，拥有此类许可证的IntelliJ IDEA客户可以免费获得一个TeamCity Enterprise Server许可证。
要检查IntelliJ IDEA许可证的TeamCity升级可用性并请求您的Enterprise Server许可证，请与销售部门联系，以获取在上述期限内购买的IntelliJ IDEA许可证之一。
