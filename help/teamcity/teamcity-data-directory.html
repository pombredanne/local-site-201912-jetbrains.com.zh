TeamCity数据目录

TeamCity数据目录是TeamCity服务器用于存储配置设置，构建结果和当前操作文件的文件系统上的目录。该目录是所有配置设置的主要存储，并保存对于TeamCity安装至关重要的数据。

构建历史记录，用户及其数据以及一些其他数据都存储在数据库中 。有关目录和数据库中存储的数据的说明，请参见备份说明。

请注意，在本文档和其他TeamCity材料中，目录通常称为.BuildServer 。如果您使用其他名称，请替换.BuildServer使用实际名称。

TeamCity数据目录的位置

当前使用的数据目录位置可以在“ 管理” | “ 管理”中查看。正在运行的TeamCity服务器实例的“ 全局设置”页面。单击“ 浏览”链接可打开“ 管理” | “ 管理”。全局设置|浏览数据目录选项卡，允许用户上载新文件/修改目录中的现有文件。

当前数据目录的位置也可以在logs/teamcity-server.log文件（在服务器启动时查找“ TeamCity Data Directory： ”行）。

如果要进行升级，请注意，在TeamCity 7.1之前，可以以其他方式指定数据目录。

配置位置

有几种方法可以配置TeamCity数据目录的位置：

  • 在第一个服务器启动屏幕上（仅当TeamCity .tar.gz要么.exe使用分布）。然后将指定的数据目录保存到< TeamCity home directory >/conf/teamcity-startup.properties文件。如果使用以下选项之一配置了数据目录位置，则不会显示用于指定数据目录位置的屏幕。

  • 手动使用TEAMCITY_DATA_PATH环境变量。该变量可以是系统范围的变量，也可以是为其启动TeamCity服务器的用户定义的变量。设置/更改变量后，您可能需要重新启动计算机才能使更改生效。

  • 如果TEAMCITY_DATA_PATH未设置环境变量，并且< TeamCity home directory >/conf/teamcity-startup.properties文件也没有定义它，默认的TeamCity数据目录位置在用户的主目录中（例如，它是$HOME/.BuildServer在Linux和%USERPROFILE%.BuildServer在Windows下）。

在TeamCity 9.1之前 ，TeamCity Windows安装程序在安装过程中通过设置以下内容来配置TeamCity数据目录： TEAMCITY_DATA_PATH环境变量。建议的目录默认路径为： %ALLUSERSPROFILE%JetBrains\TeamCity 。在更高的TeamCity版本中，安装程序不要求提供TeamCity数据目录，可以在第一次TeamCity启动时对其进行配置。

有关选择数据目录位置的建议

由于数据目录存储所有服务器和已配置的项目设置，因此，如果没有相应的访问级别，则不可用于读取和写入OS用户，这一点很重要。请参阅相关的安全说明 。

默认情况下， system该目录在历史记录中存储了构建的所有工件和构建日志，并且可能很大，因此建议将TeamCity Data Directory放置在非系统磁盘上。请参阅“ 清理”页面以配置较早版本的自动清理。如果单个本地磁盘不能存储所有工件，则可以添加另一个磁盘并配置多个工件路径 。

请注意，TeamCity假定对TeamCity数据目录具有可靠且持久的读/写访问权，并且在无法访问数据目录时可能会发生故障。当目录不可用时，此故障可能会影响TeamCity的运行，还可能破坏当前正在运行的内部版本的数据。尽管TeamCity可以容忍偶尔的数据目录不可访问性，但在极少数情况下，仍会损坏目录中存储的数据并使其部分丢失。

建议存放< TeamCity Data Directory >/system/caches （通过从主目录符号链接到本地磁盘），甚至可以是单独的专用磁盘，尤其是当TeamCity Data Directory位于网络存储上时。该目录存储了VCS存储库的本地克隆，因此必须为其提供良好的性能。如果目录内容丢失，将对其进行重建，并且不会丢失任何数据。

数据目录不应位于TeamCity主目录下 。

强烈建议使用TeamCity安装目录之外的“数据目录”位置，以防止任何数据丢失。

TeamCity数据目录的结构

的config TeamCity数据目录的子目录包含TeamCity项目的配置，以及system子目录包含构建日志，工件和数据库文件（如果使用内部数据库（HSQLDB），则为默认值）。您还可以查看有关“ 手动备份和还原”的信息，以更好地了解哪些数据存储在数据库中以及哪些数据存储在文件系统中。

  • BuildServer/config –用于存储项目，构建配置和常规服务器设置的目录

    • trash –备份已删除项目的副本，可以手动删除它们。有关恢复项目的详细信息，请参阅“ 如何...”。

    • notifications –通知模板和通知配置设置，包括联合供稿模板

    • logging – 内部服务器日志记录配置文件，可以将新文件手动添加到目录中

    • projects –包含所有与项目相关的设置的目录。每个项目都有其自己的目录。未使用项目层次结构，所有项目都有一个直接位于“项目”下的相应目录

      • –包含项目所有设置的目录，其中包含 ID（包括构建配置设置，但不包括子项目设置）。可以创建新目录，只要它们具有必需的嵌套文件即可。根目录包含根项目的设置。每当*.xml.N文件位于目录下，它们是通过Web UI更改项目配置时创建的相应文件的备份副本。 TeamCity不使用这些备份副本。

        • buildNumbers –包含以下内容的目录.buildNumbers.properties文件，用于存储对应的内部版本配置的当前内部版本号计数器

        • buildTypes –具有的目录.xml具有相应的构建配置或模板设置的文件

        • pluginData –用于存储可选的和与插件相关的项目级别设置的目录。捆绑的插件设置和辅助项目设置（如自定义项目选项卡）存储在目录中的plugin-settings.xml文件中。根据版本设置存储在VCS之外的凭据存储在secure/credentials.json文件

        • vcsRoots –在文件中包含项目的VCS根目录设置的目录.xml

        • project-config.xml –包含项目设置（例如参数和清理规则）的项目配置文件。

    • main-config.xml –服务器范围的配置设置

    • database.properties –数据库连接设置，请参阅“ 设置外部数据库”中的更多信息

    • license.keys –一个文件，该文件存储输入到TeamCity中的许可证密钥

    • change-viewers.properties – 外部更改查看器配置属性（如果有）

    • internal.properties –用于指定各种内部TeamCity属性的文件 。默认情况下它不存在，必要时需要创建

    • auth-config.xml –存储与服务器范围的身份验证相关的设置的文件

    • ldap-config.properties – LDAP认证配置属性

    • ntlm-config.properties – Windows域身份验证配置属性

    • issue-tracker.xml –问题跟踪器集成设置

    • cloud-profiles.xml –云（例如，Amazon EC2）集成设置

    • backup-config.xml – Web UI备份配置设置

    • roles-config.xml –角色权限分配文件

    • database.*.properties –不同外部数据库的默认模板连接设置文件

    • *.dtd – XML配置文件的DTD文件

    • *.dist –对应文件的默认模板配置文件，不带.dist 。见下文 。

  • .BuildServer/plugins –一个可以存储TeamCity插件的目录，以便在TeamCity启动时自动加载。可以将新插件添加到目录中。服务器不运行时，可以删除现有服务器。插件的结构在插件包装中进行了描述。

    • .tools –创建此目录以集中所有工具上要安装的工具。任何文件夹或.zip该文件夹下的文件将分发给所有代理，并显示在代理主目录文件夹下。

  • .BuildServer/system –存储构建结果数据的目录。该目录的内容由TeamCity生成，并不用于手动编辑。

    • artifacts –构建文件，日志和其他数据的默认目录 。工件存储的格式为// （了解有关内部内部版本ID的更多信息）。如有必要，可以手动添加/删除每个构建目录中的文件-这将反映在相应构建的工件中。

      • .teamcity子目录存储构建的隐藏工件和构建日志（请参见下文）。可以根据需要手动删除文件，但是不建议这样做，因为构建将丢失文件支持的相应功能（例如构建日志，显示/使用完成的构建参数，包括作为快照依赖项的构建重用，覆盖率报告） ， 等等）

        • logs子目录以内部格式存储构建日志 。构建日志存储构建输出，编译错误，测试输出和测试失败的详细信息。如有必要，可以手动删除文件，但是相应的构建将丢失构建日志和失败详细信息（以及测试失败详细信息）。

    • messages –在TeamCity 9.0之前的版本中，以前存储构建日志的目录。自动构建将日志迁移到工件下的新位置后，该目录将存储无法移动的文件（有关详细信息，请参阅服务器启动上的服务器日志）。

    • changes – 远程运行更改以内部格式存储的目录。目录内文件的名称包含内部个人更改ID。可以根据需要手动删除文件，但是相应的个人构建将丢失UI中的个人更改，并且当受影响的排队构建尝试启动时，它们将失败或在没有个人补丁的情况下运行。

    • pluginData –一个目录，用于存储有关构建，当前系统状态等的各种数据。不建议删除或修改此目录。 （例如，在TeamCity 2018.2之前，构建触发器的状态存储在此目录中）。该目录的内容与数据库中存储的数据相对应，因此，在还原数据库时，应将该目录还原到与数据库一致的相同状态。

    • audit –保存构建配置更改的历史记录的目录，用于显示更改的差异。还将相关数据存储在数据库中。

    • repositoryStates –在TeamCity 2018.2之前，它用于存储VCS根目录的当前状态，该状态已在2018.2。中移至数据库。如果删除，则TeamCity可能无法检测到某些更改（在TeamCity上次查询的状态与没有此数据的第一个服务器启动后的当前状态之间）。

    • caches –具有内部缓存的目录（VCS存储库内容，搜索索引等）。可以手动将其删除以清除缓存：它们将根据需要自动恢复。在服务器未运行时删除目录更安全。

      • .unpacked –自动创建的目录，用于存储解压缩的服务器端插件。服务器运行时不应修改。如果服务器未运行，则可以安全删除。

    • buildserver.* –与嵌入式HSQLDB有关的一组文件。

  • .BuildServer/backup –默认目录，用于存储通过Web UI创建的备份档案。 TeamCity不使用此目录中的文件，如果已经复制它们进行保管，则可以安全地删除它们。

  • .BuildServer/lib/jdbc – TeamCity用于搜索数据库驱动程序的目录 。如有必要，创建目录。TeamCity不管理目录中的文件，它仅扫描该目录以查找存储必要驱动程序的.jar文件。

直接修改配置文件

下的文件config目录可以手动编辑（除非明确注明）。在不重新启动服务器的情况下将考虑更改。TeamCity监视这些文件的更改，并在检测到修改或新文件时自动重新读取它们。请记住，很容易破坏这些文件的物理或逻辑结构，因此在编辑时要格外小心。进行任何更改之前，请始终备份数据。

请注意，文件格式可能会随TeamCity的更新版本而变化，因此文件更新过程可能需要在升级后进行调整。

REST API具有用于大多数常用设置编辑的方式，并且在服务器升级后的功能方面更加稳定。

.dist模板配置文件

许多用于手动编辑的配置文件使用以下约定：

  • 与文件一起（假设命名为fileName ）有一个文件fileName.dist 。 .dist文件用于存储默认服务器设置，因此您可以将其用作示例fileName组态。的.dist文件不应该手动编辑，因为它们在每次服务器启动时都会被覆盖。也， .dist服务器升级期间使用文件来确定是否fileName用户修改了文件，或者可以更新后者。

XML结构和参考

如果您打算手动修改配置，请注意，有些条目由id链接。这样的条目的示例是构建配置-> VCS根链接和项目->父项目链接。相同类型的所有条目在整个服务器中必须具有唯一的ID。仅当新条目的ID是唯一的时，才能添加新条目。
