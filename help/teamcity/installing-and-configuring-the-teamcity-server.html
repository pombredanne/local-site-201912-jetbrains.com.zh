将TeamCity安装到现有的J2EE容器中

不建议使用.war发行版。请改用TeamCity .tar.gz发行版（与Tomcat Web服务器捆绑在一起）。如果您出于重要原因将TeamCity部署到现有的Web服务器中并想使用.war发行版，请告诉我们原因。请参阅支持的平台和环境以了解J2EE容器要求。

  1. 确保您的Web应用程序服务器已停止。

  2. 复制下载的TeamCity.war文件放入J2EE容器的Web应用程序目录下， TeamCity.war名称（文件名通常用作URL的一部分）或根据Web服务器的文档部署.war。确保没有部署TeamCity的其他版本（例如，不要将旧的TeamCity Web应用程序目录保留在Web服务器应用程序目录下）。

  3. 确保TeamCity Web应用程序获得足够的内存量。如果您在同一JVM中运行其他Web应用程序，则相应地增加内存。

  4. 如果要将TeamCity部署到Tomcat容器，请添加useBodyEncodingForURI="true"归因于主体Connector服务器中的标签Tomcat/conf/server.xml文件。

  5. 如果要将TeamCity部署到版本大于7.5.5（包括8.xx）的Jetty容器，请确保系统属性org.apache.jasper.compiler.disablejsr199被设定为true

  6. 确保已将servlet容器配置为解压缩已部署的war文件。尽管对于大多数servlet容器，这是默认行为，但对于某些servlet容器，则不是默认行为（例如，Jetty版本> 7.0.2），应进行显式配置。TeamCity无法使用打包的.war进行工作：如果以此方式启动，则在日志和UI上会有注释。

  7. 配置适当的TeamCity数据目录以供TeamCity使用。请注意，建议从空的TeamCity数据目录开始。完成安装并执行第一个TeamCity服务器启动后，可以将所需的数据（例如， 数据库设置文件）移至该目录。

  8. 通过指定以下内容检查/配置TeamCity 日志记录属性 log4j.configuration和teamcity_logs内部属性。

  9. 通过servlet容器管理界面重新启动服务器或部署应用程序，并访问http：// server：port / TeamCity / ，其中TeamCity是的名称war文件。

TeamCity服务器需要 JRE 8才能运行。推荐的Java发行版是AdoptOpenJDK 。除非您需要为TeamCity服务器分配更多的内存 ，否则我们建议使用32位安装。升级之前，请检查64位Java注释 。如果您已配置任何本机库供TeamCity使用（例如.dll （使用Microsoft SQL数据库的Integrated Security选项），则需要更新库以匹配JVM x86 / x64平台。

使用64位Java运行TeamCity Server

TeamCity服务器可以在32位和64位JVM下运行。建议使用32位JVM，除非您需要专用于1Gb以上的内存（通过-Xmx JVM选项）添加到TeamCity进程（请参阅详细信息 ），或者您的数据库要求不同。

如果选择使用64位JVM，请注意，从32位JVM切换到64位JVM时，内存使用量几乎翻了一番，因此请确保指定的内存至少是32位JVM的两倍。 为TeamCity Server设置内存设置 。

要更新到64位Java ：

  • 更新服务器要使用的Java

  • 设置JVM内存选项 。建议为64位JVM设置以下选项：-Xmx4g -XX:ReservedCodeCacheSize=350m

设置TeamCity Server的内存设置

TeamCity服务器具有主进程，该主进程也可以启动子进程。子进程使用计算机上的可用内存，此部分仅在需要特殊配置的情况下介绍TeamCity主服务器进程的内存设置。
作为JVM应用程序，TeamCity主服务器进程仅利用专用于JVM的内存。所需的内存可能取决于所使用的JVM（32位或64位）。JVM使用的内存通常包括：堆（通过-Xmx ）和元空间（受可用本机内存的数量限制），内部JVM（通常为数十Mb）以及与操作系统相关的内存功能（如内存映射文件）。TeamCity的主要取决于堆存储器，该设置可以通过手动被配置为应用TeamCity的传递 -Xmx运行TeamCity服务器的JVM的（堆空间）选项。

一旦开始将TeamCity用于生产目的或要在评估期间加载服务器，则应手动为TeamCity服务器设置适当的内存设置。

要更改内存设置 ，请参阅配置TeamCity服务器启动属性 ，或者如果使用以下命令运行TeamCity，则请参阅应用程序服务器的文档： .war分配。
通常这意味着设置TEAMCITY_SERVER_MEM_OPTS环境变量的值如-Xmx750m

我们建议删除-XX:MaxPermSize的JVM选项TEAMCITY_SERVER_MEM_OPTS环境变量（如果先前已配置），因为在Java 8中将其忽略。

如果速度慢，发生OutOfMemory错误，或者在TeamCity UI中始终看到与内存有关的警告，请将设置增加到下一个级别。

  • 最小设置 （应使用32位Java（捆绑在.exe分发中））：-Xmx750m

  • 中型服务器使用的推荐设置 （应使用32位Java）： -Xmx1024m 。使用32位Java进行更大的设置可能会导致OutOfMemoryError，并显示“本地内存分配（malloc）失败” JVM崩溃或“无法创建新的本机线程”消息

  • 大型服务器使用的建议设置 （应使用64位Java ）： -Xmx4g -XX:ReservedCodeCacheSize=450m 。这些设置应适用于最多具有两百个代理和数千个构建配置的安装。安装的自定义插件可能需要增加通过Xmx参数定义的值。

  • 大规模服务器使用的最大设置 （应使用64位Java ）： -Xmx10g -XX:ReservedCodeCacheSize=640m 。较大的值可用于较大的TeamCity安装。但是，通常不建议在未咨询TeamCity支持的情况下使用大于10g的值。

温馨提示 ：

  • 32位JVM可以可靠地使用多达1Gb的堆内存（ Xmx1024m ）。（可以增加到-Xmx1200m ，但使用此设置，Windows下的JVM有时可能会崩溃。）如果需要更多内存，则应使用分配了不少于2.5Gb的64位JVM（ -Xmx2500m ）。除非TeamCity服务器与100多个代理一起运行或为非常活跃的内部版本/数千名用户提供服务，否则不太可能需要为TeamCity进程分配超过4Gb的内存。

  • 一条经验法则是，为同一应用程序分配的64位JVM的内存应是32位的两倍。如果切换到64位JVM，请确保调整内存设置（ -Xmx ）。如果您为应用程序分配的内存少于两倍，则切换到64位是没有意义的。

  • 大型TeamCity安装可能会受益于内存设置的微调。专用于TeamCity服务器JVM的内存量不应定期超过计算机上总可用物理内存的60％（以允许嵌套进程和OS级别的缓存使用）。还有，用堆（ –Xmx ）设置为大于8Gb，如果计算机具有许多CPU内核（例如，超过8个）并且当前CPU使用率低于60％，则可以通过以下方式启用G1 JVM垃圾回收器： -XX:+UseG1GC JVM选项可能会减少世界各地GC暂停的时间。

推荐的方法是从初始设置开始，然后使用“ 管理” | “ 管理”来监视已用内存的百分比。 诊断页面。如果服务器在数十分钟内始终使用80％以上的内存而不会出现中断，则可能是增加内存的迹象。 -Xmx内存值再降低20％。

设置管理帐户后，您可以开始在TeamCity服务器中创建项目和构建配置。您可能还需要在“服务器管理”部分中配置以下设置：

  • Jabber服务器地址和设置

配置服务器以供生产使用

现成的TeamCity服务器安装适用于评估目的。对于生产用途，您将需要执行其他配置，通常包括：

  • 检查服务器是否正在使用适当的服务器端口，并通过https配置访问 。

  • 确保服务器URL，电子邮件和（可选）Jabber服务器设置已指定且正确

  • 配置服务器进程以在机器重新引导时进行与OS相关的自动启动

  • 为TeamCity数据目录使用可靠的存储

  • 使用外部数据库

  • 配置建议的内存设置 ，对活动或增长中的服务器使用“最大设置”

  • 规划定期备份

  • 计划定期升级到最新的TeamCity版本

  • （自TeamCity 10.0.3起）考虑添加teamcity.installation.completed=true行到< TeamCity数据目录 > \ conf \ teamcity-startup.properties文件中–如果找不到这样的用户，这将阻止服务器创建管理员用户
