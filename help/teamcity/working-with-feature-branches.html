使用功能分支

分布式版本控制系统（DVCS）中的功能分支使您可以独立于主开发而开发功能，并将该功能的所有更改提交到该分支上，并在完成功能后将这些更改合并到主分支中。这种方法为软件开发团队带来了许多优势。但是，在没有专用支持的连续集成服务器中，它也会引起许多问题，例如持续的内部配置重复，可见性差以及最终失去对过程的控制。

TeamCity对功能分支的支持在不断增加，除其他功能外，TeamCity还包括分支远程运行触发器，每次TeamCity在构建配置的VCS根目录的特定分支中检测到更改时便启动新的个人构建，并自动合并以将一个分支合并到另一个分支中成功构建后。

支持的版本控制系统

支持Git和Mercurial功能分支以及Perforce 分支流支持 。

配置分支

要开始使用DVCS分支，您需要配置需要监视哪些分支的更改。这是通过“ 分支规范”字段在Git或Mercurial VCS根目录的“ 常规设置”部分中完成的。
使用Perforce，请选中相应的框以启用功能分支支持，这将显示分支规范字段。该字段接受分支名称或模式的列表。除默认分支外，TeamCity还监视与分支规范匹配的分支 。

配置分支规范后，TeamCity将开始监视这些分支的更改。如果您的构建配置具有VCS触发器，并且在某个分支中找到更改 ，则TeamCity将在该分支中触发构建。在构建配置主页上，您还可以通过分支名称过滤历史记录，更改日志，待处理的更改和发布日志。分支名称也将出现在自定义构建对话框中，因此您也可以在分支上手动触发自定义构建。

分支规范字段的语法是换行符分隔的“ +|-:branch_name ”规则。

在这里， 管道符号|代表OR命令，如正则表达式中一样：use +用于包括， 或 -排除。

每个规则可以有一个可选*匹配一个或多个字符的占位符。
例如， +:refs/heads/teamcity*匹配所有以开头的分支refs/heads/teamcity 和至少一个其他字符 。
与分支refs/heads/teamcity将不匹配。

的branch_name参数是特定于VCS的，即refs/heads/master在Git中：

分支名称中由星号（ * ）通配符成为要在TeamCity用户级界面中显示的短分支名称（也称为逻辑分支名称 ）。该行还可以包含可选的括号，当出现时，它们表示要用作逻辑名称的模式部分，而不仅仅是*匹配的符号。

您可以在分支规范中使用参数。

当单个VCS分支与分支规范的多行匹配时，将应用最具体的（最小字符与模式匹配）最后一个规则。

也就是说，如果规范包含与分支完全匹配的模式（即不包含*通配符），则使用最后一个这样的模式。因此，如果您有这样的规范：

+：refs / heads / release-v1-：refs / heads / release-v1

那么最后一个模式将获胜，分支将被排除。

如果分支规范具有多个模式， *通配符，然后TeamCity选择产生最短逻辑名的模式。该分支规范：

+：refs / heads / * / hotfix-：refs / heads / v1 / *

将包括refs/heads/v1/hotfix分支（因为“ v1 “比”短hotfix ”）。
如果有2种图案*通配符产生相同长度的逻辑名称，然后最后一个模式获胜。

分支规范支持以“”开头的行作为注释#" 。

还有一种特殊的转义语法，通过#! escape: CHARACTER语法：例如，要在分支名称中使用圆括号，则需要对其进行转义。假设您要追踪release-(7.1) branch：为此，请在规范的第一行中指定一个转义符号。对于Mercurial，以下分支规范可以做到这一点：

＃！转义：\ +：release-\（7.1 \）

要在GitHub Pull Request分支上运行构建，请使用Pull Requests构建功能。

默认分支

为DVCS配置VCS根目录时，需要指定要用作默认名称的分支名称。默认分支具有特殊含义：

  • 当未指定分支或分支规范未包含指定的分支时（例如，当某人仅单击“ 运行”按钮而未选择分支时），将使用此后备分支。

  • 当显示构建和更改序列并到达分支创建时刻时，可以使用它。

  • 默认分支允许在不同的VCS根目录中使用不同的分支（例如，如果其中一个根目录是Git，而另一个根目录是Mercurial），并且在通过快照依赖项进行链接时可以在不同的版本中使用不同的分支。当在默认分支中触发顶级链构建时，其所有依赖项也将在其各自的默认分支中构建。

除非通过分支过滤器禁用，否则默认分支始终隐式包含在分支规范中。在TeamCity UI中，默认分支用分支标记的较深背景标记。

我的分支

TeamCity可以根据当前TeamCity用户的提交来标识分支并将其分组。

在分支过滤器中选择“ 我的分支”组，以根据定义的VSC用户名显示其最近100次更改包括您的提交的所有活动分支。

逻辑分支名称

逻辑分支名称是在用户界面中针对构建和构建配置级别显示的分支名称。逻辑分支名称通常是完整的VCS特定分支名称的一部分。通过将分支规范应用于版本控制中的分支名称来计算。

例如，如果分支规范是这样定义的：

+：裁判/负责人/ *

然后匹配的部分* （例如， master ）是逻辑分支名称。

如果分支规范模式使用括号，则逻辑名称由括号内名称的一部分组成；看到v8.1/feature1在VCS分支的UI中显示的逻辑名称refs/heads/v8.1/feature1 ， 用这个：

+：裁判/负责人/(v8.1/*）

您无需将默认分支包括在分支规范中，因为它已经隐式包含在那里。但是，例如，如果您想为UI中的默认分支使用一些简短的逻辑分支名称， master ，您可以将其包含在分支规范中并使用括号：

+：裁判/负责人/（大师）

变化

对于每个构建，TeamCity都会显示其中包含的更改。对于来自分支的构建，更改计算过程将考虑分支，并向您展示与构建分支相关的更改。分支中某个版本的更改计算为从该版本的修订版到该分支中先前版本或默认分支中的版本的更改。
带有提交图的更改日志将帮助您了解受监视分支中的情况。

默认情况下，启用“ 显示图形”选项时，TeamCity在图形上显示构建标记。

活动分支

在具有已配置分支的构建配置中，默认情况下，大多数UI页面都会显示活动分支。

活动分支是具有最近活动的分支：它具有最新版本，或存在于具有最新提交的存储库中。

活动阈值可以通过构建配置参数进行配置。可以在构建配置（这只会影响一个构建配置），项目或内部属性 （这定义整个服务器的默认值）中更改参数。配置中的参数将覆盖内部属性中的参数。

在以下情况下，分支被认为是活动的：

  • 它存在于VCS存储库中，并且具有最近的提交（即，其提交时间小于VCS的值） teamcity.activeVcsBranch.age.days参数，默认为7天）。

  • 或它具有最近的版本（即年龄小于的值的版本teamcity.activeBuildBranch.age.hours参数，默认为24小时）。
    上次构建后的24小时内，具有构建的已关闭VCS分支仍将显示为活动状态。要从显示中删除关闭的分支，请设置teamcity.activeBuildBranch.age.hours=0 。

测验

TeamCity尝试检测构建中的新失败测试，对于那些不是新的测试，您可以看到测试在哪个构建中开始失败。此功能也知道分支，即，在计算第一个构建时，TeamCity遍历同一分支的构建。

此外，“测试详细信息”页面上提供了分支过滤器 ，您可以在单个分支中查看测试通过或失败的历史记录。

失效条件

如果按以下方式配置了构建失败条件： 与上次成功/完成/固定的构建相比，构建度量标准已更改 ，则将使用来自同一分支的构建。如果在同一分支上没有合适的构建，则使用默认分支构建，并将相应的消息添加到构建日志。

扳机

VCS触发器完全了解分支，一旦在分支中检测到签入，就会触发构建。所有VCS触发选项（如每个签入触发，静默期和触发规则）都可直接用于分支机构的构建。默认情况下，“计划和完成”构建触发器将仅在默认分支中监视构建。

此外，可以为VCS，计划和完成构建触发器指定分支过滤器 。

依存关系

如果具有分支的构建配置具有与具有分支的其他构建配置的快照依赖关系，则当触发分支中的构建时，如果构建的VCS根目录中的分支具有分支，则链中的其他构建也将获得关联的分支。 逻辑名称相同，且分支规范未排除此分支。构建的VCS根目录可以指向不同的存储库，但是逻辑分支名称必须相同。

如果满足此条件，则将检出具有该名称的分支，并将链下的所有构建（触发构建所依赖的构建）和链上的所有构建（取决于触发构建的）标记为相同科。否则，默认分支将被检出。

可以配置构件依赖项以从特定分支的构建中检索构件：构件依赖将使用来自指定分支的构建。计划和完成构建触发器也是如此。

通知事项

除“我的更改”外，所有通知规则仅会在默认分支的构建中通知您。同时，“我的更改”规则将适用于所有可用分支中的构建。

构建配置状态

仅基于默认分支中的构建来计算构建配置状态。因此，按配置调查适用于默认分支中的构建。例如，从非默认分支成功构建将不会删除每个配置的调查，但从默认分支成功构建将不会删除。

多个VCS根

如果您的构建配置使用多个VCS根目录，并且您指定了要在两个VCS根目录中监视的分支，则触发构建的方式会更加复杂。

VCS触发器组通过逻辑分支名称从多个VCS根分支分支 。当某个根没有另一个根的分支时，将使用其默认分支。例如，您有2个VCS根，都具有默认分支refs/heads/master ，第一个根具有分支规范refs/heads/7.1/*和分支机构的变化refs/heads/7.1/feature1和refs/heads/7.1/feature2 ，第二个根具有规范refs/heads/devel/*和分支的变化refs/heads/devel/feature1 。在这种情况下，VCS触发器将运行3个版本，并具有以下分支组合的修订版：

根1

root2

裁判/负责人/大师

裁判/负责人/大师

裁判/头/7.1/功能1

裁判/头像/开发/功能1

裁判/头/7.1/功能2

裁判/负责人/大师

构建参数

如果您需要在构建脚本中获取分支名称或在其他构建配置设置中将其用作参数，请参阅预定义的构建参数 。

清理

清理规则独立应用于每个活动分支 。

手动分支合并

您可以在TeamCity中手动合并分支，例如，如果您只想在代码审查/批准后合并分支，或者即使测试失败在分支中也要执行合并。

手动合并源 ：

打开构建结果页面 ，单击右上角的“ 操作”菜单，然后选择“ 合并此构建源 ”。
出现的对话框使您可以选择目标分支并添加提交消息（必填）。
