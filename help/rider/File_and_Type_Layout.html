文件和类型布局

JetBrains Rider可以根据不同的模式对C＃文件中的类型进行重新排序和键入成员。模式可以描述许多条件和约束，这些条件和约束在对文件中的项目重新排序和/或将它们包装在指定区域中时进行评估。

您的文件布局首选项是使用基于图层的设置机制保存的。除其他外，此机制使您可以针对不同的解决方案维护不同的首选项，并将这些首选项保持在VCS下，并自动与团队成员共享。

布局模式简介

您可以配置多个布局模式以应用于不同的上下文（例如，您可以为类和接口使用不同的模式）。

在模式中，可以按所需顺序指定匹配项，也可以指定将匹配项包装/分组到的区域/组。

应用当前模式集的算法如下：

  1. 如果存在文件模式，JetBrains Rider会检查是否应删除文件中的区域，然后会检查模式中的每个匹配项是否与文件中的任何代码项匹配。

  2. 如果存在匹配条目，则它们在文件中的位置将根据文件模式中相应匹配项的位置进行更改。

  3. 如果同一代码项与多个匹配器条目匹配，那么将应用优先级更高或约束更强的匹配器。也就是说，如果有匹配的公共类型和匹配公共静态类型，那么所有的公共静态类型将由图案B匹配模式B模式A.要更改此设置，可以提高模式A的优先级。

  4. 如果存在具有相同约束集的匹配器条目，则根据第一个此类匹配器条目的位置移动匹配项。

  5. 如果指定了区域或组，则将匹配的项目相应地分组或包装在区域中。
    组或区域允许您为其指定优先级。例如，如果某个组/区域比其外部的其他匹配器具有更高的优先级，则JetBrains Rider首先处理该组/区域内的匹配器，然后再处理其余的匹配器。组区域之间的唯一区别是该区域将匹配的项目包装到#region / #endregion 。

  6. 所有不匹配的内容都将移到匹配项之后。如果需要将不匹配的项目放在其他位置，则可以创建一个没有限制的匹配器，并在模式中为其设置所需的位置。

  7. 此后，将应用类型模式（如果有）。JetBrains Rider检查每种类型模式是否与文件中的任何类型匹配。如果某个类型有多个匹配模式，则与步骤3和步骤4类似地解决冲突。

  8. 建立类型匹配后，JetBrains Rider会检查是否应删除类型中的区域，然后会检查模式中的每个匹配器条目是否与文件中的任何成员匹配。

  9. 如果存在匹配成员，则它们在类型中的位置会根据相应匹配项在类型样式中的位置而改变。

  10. 区域，组和不匹配成员的处理与步骤5和步骤6相似。

文件和类型布局首选项

默认的成员布局规则基于众多最佳实践，在大多数情况下都可以建议使用。但是，如果您的个人喜好或公司标准与JetBrains Rider的默认设置不同，则可以基于默认设置配置代码重新排序模式，并为特定情况创建新的模式。
