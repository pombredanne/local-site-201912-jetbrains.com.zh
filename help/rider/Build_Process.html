建立过程

要构建当前解决方案，请按Ctrl+Shift+B 。或者，可以从“构建”菜单开始构建，重建或清理。您还可以通过自定义方式来构建特定项目，例如，使用诊断或不依赖项-在解决方案资源管理器中选择所需的项目，然后使用上下文菜单中“ 高级构建操作”组下的操作 。

运行构建时，“构建工具”窗口中将显示MSBuild或XBuild输出Alt+0 。完成后，窗口将显示错误和警告。

默认情况下，Rider通过 ReSharper Build ，它跟踪解决方案中的更改，并且仅重建修改后的项目和必要的依赖项。
如果您的解决方案包括非MSBuild项目或具有自定义生成步骤的项目，则需要禁用ReSharper Build on 构建，执行，部署|的工具集和构建页面 JetBrains骑士设置Ctrl+Alt+S —在这种情况下，Rider会将整个解决方案构建委托给MSBuild.exe 。

ReSharper Build如何工作

首次运行ReSharper Build时，它将根据您的构建配置使用多个并行进程来构建所有项目。

下次构建解决方案时，JetBrains Rider仅构建更改了支持的构建项目的项目 。如果更改影响其他项目中使用的公共类型，则也会重新构建从属项目。

不受更改影响的项目将被完全跳过，包括所有其他构建逻辑。如果要为跳过的项目执行构建前和构建后事件和/或BeforeBuild和AfterBuild MSBuild目标 ，请使用以下相应的复选框 构建，执行，部署|的工具集和构建页面 JetBrains骑士设置Ctrl+Alt+S 。

ReSharper Build维护MSBuild任务，目标和项目的输入和输出的依存关系图，并有效地监视文件系统的更改。当您开始构建时，ReSharper Build已经知道时间戳是否是最新的，而无需调用MSBuild：如果时间戳是最新的，则将跳过该项目。

默认情况下，ReSharper Build使用最新安装的MSBuild版本，并在6个进程中并行运行。如有必要，您可以显式指定MSBuild版本并更改以下进程的数量 构建，执行，部署|的工具集和构建页面 JetBrains骑士设置Ctrl+Alt+S 。

ReSharper Build支持的项目类型和限制

ReSharper Build支持大多数基于MSBuild的.NET项目类型和语言（C＃，VB.NET，F＃，C ++），包括.NET Core项目。但是，存在以下限制：

  • 非MSBuild项目 。ReSharper Build无法用于非基于MSBuild的项目类型。这包括网站（基于文件夹的项目，而不是MVC Web应用程序），WinJS应用程序和DNX应用程序。如果您的解决方案中有这些类型的项目，则整个解决方案构建将退回到标准的MSBuild构建过程。

  • 自定义构建步骤 。通常，自定义构建步骤（包括C ++项目中的构建前和构建后步骤）等同于调用批处理文件，并执行多个命令行以将文件复制到输出目录中。就ReSharper Build而言，这是一个黑匣子-它不知道自定义构建步骤中发生了什么，因此无法跟踪输入和输出。

    但是，自定义MSBuild目标和任务将与ReSharper Build一起使用，只要自定义目标和任务支持增量构建 ，并带有可监视时间戳更改的输入和输出列表即可。强烈建议将天真的自定义构建步骤转换为自定义目标，以便通过ReSharper Build获得良好的效果。
