InspectCode命令行工具

ReSharper最著名的功能之一，即代码检查 ，即使不打开Visual Studio也可以使用。InspectCode是一种免费的命令行工具，至少需要一个参数（您的解决方案文件）才能应用ReSharper的所有检查。

要运行InspectCode：

  1. 下载ReSharper命令行工具

  2. 将命令行工具包解压缩到任何目录中。
    解压缩之前，请确保已下载的.zip文件已被“取消阻止”：右键单击该文件，选择“ 属性” ，然后单击“ 取消阻止” 。否则，将导致.NET框架以部分信任的方式加载应用程序，这意味着它将无法正确加载或运行。

  3. 构建目标解决方案。

  4. 运行以下命令：

    InspectCode.exe YourSolution.sln -o =

如果您正在使用InspectCode找到C ++代码的问题，使用inspectcode.x86.exe并且不使用inspectcode.exe

您也可以从命令行运行重复分析和覆盖率分析 。

了解输出

InspectCode完成分析后，将结果保存在命令提示符下指定的XML文件中-o= 。该文件包括在指定范围内找到的所有代码问题（整个解决方案或特定项目）。XML包含两部分：

  • 发现的问题类型列表，其中每种类型对应于特定的检查并具有以下属性：

    • Id —允许将每个问题链接到相应的检查。

    • Category -可用于按类别对相似问题进行分组。

    • SubCategory —如果某些问题类型具有相同的SubCategory属性，则表示这些问题相同，但以不同的语言或不同的范围发现。您可以将其用于进一步的分组。

    • Description -描述问题

    • Severity —显示检查的严重程度 。

    • WikiUrl —相应代码检验索引条目的链接（如果可用）。

    • Global —通知解决方案范围内的代码检查 。

  • 已发现问题按项目分组的列表，其中每个问题具有以下属性：

    • TypeId —允许将每个问题链接到相应的检查（ IssueType在报告的第一部分）

    • File —相对于解决方案的受影响文件的路径。

    • Offset —从文件的开头到有问题的代码的开头和结尾的符号偏移范围。

    • Line -包含有问题的代码的行。

    • Message —问题的简短描述。

    • Severity -这个属性才会出现，如果严重程度从相应检查的严重性问题的不同。如果解决方案中的某些项目启用了“将警告作为错误处理”选项，而其他项目则未启用，则这是可能的。在这种情况下，某些问题的严重性将不同于原始的“警告”严重性。

如何处理输出取决于您自己。但是，这里有几个建议的下一步：将输出转换为HTML报告，或者根据检测到的问题的数量和类型在持续集成（CI）服务器上生成一些消息。

获取代码问题列表的另一种方法是在ReSharper中运行代码检查 ，然后将检查结果导出到文件中 。但是，这可能会产生稍微不同的结果，因为：

  • InspectCode不包括具有提示严重性级别的问题 ，但ReSharper包括。

  • InspectCode可以使用其他配置，这些配置可能会禁用某些检查或更改其严重性级别。例如，如果您使用--profile参数以加载自定义设置。

使用场景

现在，让我们看看如何使用该工具以及如何对其输出进行精确处理。在本地计算机上运行它可能会有所帮助，但仅当您没有ReSharper时，因为使用ReSharper，您只需单击几下即可获得选定范围的检查结果，并在必要时将检测到的问题导出到报告中文件 。另外，使用ReSharper，您可以打开InspectCode报告 。

更有希望的情况是在CI服务器上使用InspectCode，您可以在其中将其集成到构建脚本中，并将代码检查结果添加到构建报告和消息中。

配置InspectCode

使用ReSharper设置配置InspectCode

如果您以前使用ReSharper处理目标解决方案，则可能已经配置了代码检查设置 。如果是这样，InspectCode将在中找到您的自定义设置 。点设置文件并应用它们。如果没有设置文件，则默认的严重性级别将用于所有检查。除了用于代码检查的自定义严重性级别之外，InspectCode还将在中查找以下设置。DotSettings文件：

  • 是否启用了解决方案范围的分析 （可以在中进行配置。DotSettings文件或带有--swea / --no-swea参数）

  • 命名规则 （只能使用进行配置。DotSettings文件）

  • 代码分析中排除的文件，文件夹和文件掩码 （只能使用进行配置。DotSettings文件）

  • 具有生成代码的文件，文件掩码和区域，其中部分禁用了代码分析功能 （只能使用进行配置。DotSettings文件）

  • 代码分析引擎应在其中存储缓存的地方。您可以在环境|环境中指定它。 ReSharper选项的常规页面或带有--caches-home命令行参数。

如果要在CI服务器上配置InspectCode，则可以使用ReSharper在本地进行所有配置， 将设置保存到“解决方案团队共享”层 ，然后提交生成的YourSolution.sln。解决方案目录中的DotSettings文件到VCS。服务器上的InspectCode将找到并应用这些设置。

或者，您可以指定共享路径。DotSettings文件（它将覆盖其他设置文件中的设置，如果有的话），通过--profile参数。

使用命令行参数配置InspectCode

上面我们已经提到了一些可选参数。这是命令行参数的完整列表（您可以通过键入以下内容进行列出） InspectCode.exe --help ）：

  • --output-o —使您可以设置输出文件。

  • --project —允许分析特定项目而不是整个解决方案。在此参数之后，您可以键入与解决方案中的多个项目匹配的项目名称或通配符。例如， --project=*Billing 。

  • --profile —默认情况下，InspectCode将使用“解决方案团队共享”层 SolutionName中的 ReSharper设置覆盖其默认设置。DotSettings （如果存在）。如有必要，可以使用此参数指定另一个。DotSettings文件，它将覆盖所有其他设置。例如，
    --profile="C:\Work\MyRsSettings.DotSettings" 。

  • --format (-f) —默认情况下，InspectCode以XML格式写入其输出。如有必要，您可以指定其他输出格式[Html, Text]使用此参数。例如， -f=Text 。

  • --jobs (-j) —默认情况下，InspectCode使用启发式方法拆分其作业，并使用尽可能多的线程/核并行运行它们。如有必要，您可以限制线程数，例如， -j=4 。

  • --absolute-paths (-a) —默认情况下，InspectCode的报告中的文件是使用相对于解决方案文件的路径写入的。您可以使用此开关在报告中具有绝对路径。

  • --swea和--no-swea -这些参数可让您显式启用或禁用解决方案范围的分析 。否则，将基于现有设置启用或禁用解决方案范围的分析。

  • --severity (-s) —默认情况下，InspectCode仅报告严重性级别为“ 建议”或更高的问题。此参数使您可以将报告的最低严重性级别更改为[INFO, HINT, SUGGESTION, WARNING, ERROR] 。例如， -s=WARNING 。

  • --debug (-d) —使用此选项可将InspectCode的执行详细信息添加到输出中。如果您在使用InspectCode时遇到问题，请与支持团队联系时使用这些详细信息。

  • --verbosity —默认情况下，InspectCode仅在日志中显示信息消息。使用此参数可以按以下级别更改写入日志的信息量（顺序从小到详细）： [OFF, FATAL, ERROR, WARN, INFO, VERBOSE, TRACE] 。
    例如，如果InspectCode出了问题，则可以联系ReSharper支持并与所有TRACE消息共享日志文件： --verbosity=TRACE 1> [path_to_log_file] 2>&1 。

  • --disable-settings-layers-dsl —禁用指定的设置图层 。接受的值： GlobalAll ， GlobalPerProduct ， SolutionShared ， SolutionPersonal

  • --no-buildin-settings —禁止使用全局，解决方案和项目设置配置文件。相当于使用--disable-settings-layers: GlobalAll; GlobalPerProduct; SolutionShared; SolutionPersonal; ProjectShared; ProjectPersonal

  • --caches-home —使您可以为InspectCode缓存的数据指定自定义位置。默认情况下，除非有设置文件，否则使用％LOCALAPPDATA％目录，在这种情况下，将使用那里指定的文件。如果要将高速SSD磁盘用于高速缓存，或者要将所有构建处理数据存储在一个位置，则此参数可能很有用。

  • --properties —使您可以覆盖MSBuild属性。您可以分别设置每个属性（ --properties:prop1=val1--properties:prop2=val2 ），或使用分号分隔多个属性--properties:prop1=val1;prop2=val2 。指定的属性将应用于所有分析的项目。当前，没有直接的方法仅将属性设置为特定项目。解决方法是在此项目中创建一个自定义属性，并将其分配给所需的属性，然后在dupFinder参数中使用该自定义属性。

  • --dumpIssuesTypes-it —使用此选项将所有现有的代码检查转储到输出中 。此选项应与实际分析分开使用，即没有解决方案参数。

  • --config-create和--config —这些选项使您可以将上述参数与配置文件一起传递。第一个选项将根据当前参数创建一个配置文件。第二个选项用于从该文件加载参数。

  • --toolset —明确指定了MsBuild Toolset版本（12.0、14.0、15.0）。例如， --toolset=12.0 。

  • --targets-for-references —将执行自定义MSBuild目标的名称，以获取引用的项目程序集。目标在项目文件或.targets文件中定义。多个值用分号分隔。例如， --targets-for-references="GetReferences" 。

  • --targets-for-items —将执行以获取项目的其他项（例如，编译项）的自定义MSBuild目标的名称。目标在项目文件或.targets文件中定义。多个值用分号分隔。例如， --targets-for-items="GetCompileItems" 。

这是带有几个命令行选项的运行InspectCode的示例：

InspectCode.exe --project = Documents -o =“ C：\ temp \ Results.xml” --no-swea -x = EtherealCode。ReSpeller“ C：\ Projects \ ReSharper \ resharper.sln”

将ReSharper扩展与InspectCode一起使用

一些ReSharper扩展提供了附加的代码检查功能 ，使您可以在代码中发现更多潜在的问题。这样的扩展名也可以与InspectCode一起使用。

使用InspectCode从ReSharper扩展运行额外的代码检查

  1. 找到包含ReSharper扩展名的NuGet包.nupkg 。您可以从ReSharper Gallery获得它，也可以按照ReSharper DevGuide中的说明为您自己的扩展程序创建它。

  2. 这.nupkg文件复制到其中InspectCode.exe所在的目录。

  3. 运行InspectCode。它将检测目录中所有带有ReSharper扩展名的NuGet软件包，并自动加载它们。
