在此页面中，您可以调整一些首选项，这些首选项会影响处理C ++代码的性能。

描述

缓存任务的最大并发

默认情况下，缓存任务的最大并发性是无限的，也就是说，ReSharper将使用所有可用的CPU内核在后台构建缓存。
如果遇到性能问题（UI响应滞后），则可能需要限制该数目，这可能与后台缓存占用太多内存有关，从而经常触发GC。

更改包含对象中的预处理程序指令后，重新索引包含者

默认情况下，更改标头中的预处理程序指令后，ReSharper不会为标头文件的包含程序重新编制索引。这有助于提高性能，但也可能导致为includer文件缓存过时的符号。
此复选框启用了重新索引编制，这使已解决的依赖关系更加精确，但可能会降低性能。

在Visual Studio 15.3和更高版本中异步读取项目属性

对于其大多数功能（包括导航 ， 代码分析 ， 重构和代码完成 ），ReSharper会构建并维护当前解决方案的模型。该模型被缓存，然后用于立即访问解决方案结构中的任何项目。

要构建此模型，ReSharper可以使用两种可用的Visual Studio API中的一种-旧的和新的，这是Visual Studio 15.3中引入的，默认情况下使用。

如果您在代码分析中遇到一些问题，则可以使用此选项恢复到较旧的API。

索引CMake和Open Folder解决方案中的独立标头

在分析CMake和打开文件夹解决方案时，ReSharper会忽略不包含在任何地方的标头。这样的解决方案可能有许多未使用的头，并且使此选项保持禁用状态以提高性能是有意义的。但是，如果解决方案结构清晰，并且很快将要使用几个标题，则可以启用此选项以使用这些标题中的ReSharper功能。

ReSharper的设置保存在设置层中，使您可以为不同的解决方案使用不同的设置，并与团队共享您的设置。要了解更多信息，请参阅管理和共享ReSharper设置 。
