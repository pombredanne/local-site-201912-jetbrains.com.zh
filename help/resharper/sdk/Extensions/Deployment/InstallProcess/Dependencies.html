依存关系

安装ReSharper平台时，所有二进制文件均以NuGet软件包的形式提供。由JetBrains创建的第一方二进制文件以可反映功能逻辑分组的软件包提供，例如JetBrains.Platform.Core.Text提供用于文本的程序集，提供用于处理文档和文本范围的界面。

这些NuGet软件包不是传统的NuGet软件包-像扩展软件包一样，它们不打算添加到项目中以提供编译参考。相反，它们旨在传递将在运行时使用的代码。这意味着软件包中文件的布局不符合传统的lib文件夹结构。正在部署的所有文件都位于DotFiles文件夹，采用扁平结构-这些文件将被复制到安装目标文件夹。为了加载依赖的程序集，文件采用平面文件夹结构-所有程序集都直接存在于DotFiles ，并且不在任何子文件夹中（尽管程序包确实包含子文件夹，但通常用于数据文件。扩展程序将其用于设置和外部注释。

ReSharper平台还利用各种第三方程序集（例如Json.NET ）来解析JSON。理想情况下，安装程序可以直接从nuget.org上的NuGet软件包中直接使用这些程序集。

但是，nuget.org软件包被设计为在编译时使用，并且可以包含程序集的多个实例，以不同的.Net Framework版本或平台为目标。ReSharper不需要程序集的多个副本-必须对其进行编译和测试的版本是该版本。ReSharper不会让安装程序在选择要使用的正确副本时复制NuGet的功能，而是会创建包含所需依赖项的新软件包。此外，新包装中的组件DotFiles文件夹，格式与第一方组装包相同。

此外，ReSharper使用一些没有NuGet软件包的第三方程序集。这些也将重新打包以适合部署模型，并使程序集可用。它们不打算在代码项目中作为NuGet包引用使用。

利用第三方依赖

创建扩展时，在添加对第三方程序集的依赖项时应格外小心。如果可能的话，最好（也是更简单）避免这种依赖关系-仅在无法轻松将功能添加到扩展的源代码中时才采用依赖关系。

扩展程序不应将第三方程序集作为其自身包装的一部分提供。

包，第一方或扩展名中的所有文件均从DotFiles包中的“文件夹”到同一安装文件夹。如果多个软件包试图分发同一第三方程序集，则安装程序将尝试将所有版本复制到同一位置。结果是不确定的，但是可能导致安装了错误的版本（较旧或较新），并且一个或多个扩展（或产品本身）可能变得不可用。

如果扩展程序希望使用第三方程序集，则应创建一个仅包含该程序集的新程序包，该程序包位于DotFiles文件夹，然后将其推送到ReSharper Plugin Gallery 。程序包的名称应与程序集的名称相同，并且程序包的版本应反映程序集的版本（或程序集的nuget.org程序包的版本）。通过使用可预测的名称和版本，其他扩展也可以使用相同的软件包。

扩展本身应该在扩展的.nuspec文件。例如

... id="Wave" version="[2.0]" /> id="Foo.Bar.ThirdParty" version="[1.3, 1.4)" /> ...

如果一个以上的扩展程序需要相同的组装程序包，则安装程序可以使用NuGet的版本解析来查找可以由两个扩展程序使用的通用版本。最好的做法是在.nuspec包含版本范围的文件，以提供最大的灵活性。

传递依存关系

扩展需要自己指定所有传递依赖.nuspec文件。也就是说，如果需要扩展包A和包装A反过来需要包装B ，则扩展名应列出两个软件包A和B作为其依赖.nuspec文件。

这是由于用于分发ReSharper扩展的NuGet画廊软件中的技术限制所致。所有扩展程序包都上载到同一画廊，无论它们针对的是ReSharper版本。当ReSharper从图库中请求已知扩展的列表时，它是在“精选提要”上进行的。该精选的供稿是画廊中所有包装的自动维护的子集。上传扩展程序时，策展人将检查软件包的依赖关系，并将其放入适当的Feed中。例如，如果依赖项是Wave=[2.0] ，策展人会将包裹放入Wave_v2.0饲料。

策展人还将尝试将依赖项添加到Feed中。但是，为简单起见，策展人仅将直接依赖项添加到提要中，并且包必须首先存在于图库中。如果不存在依赖项，则扩展安装将失败。

由于可以通过一定范围的软件包版本来满足软件包的依赖关系，因此策展人必须将所有兼容版本的依赖关系添加到提要中。如果处理了传递性依赖关系，则策展人还必须添加满足原始扩展所要求的所有软件包版本的依赖关系的所有软件包的所有版本，等等。为了简单起见，策展人只处理直接的依赖关系，而不是处理可能潜在的大量软件包（并处理无效的依赖关系周期和版本号冲突）。

建议是：

  • 扩展应该自己指定所有传递依赖项.nuspec夹。
  • 在扩展包之前，所有依赖项都应推送到库。
  • 如果有任何问题，请提出问题或发布到Google网上论坛 。
