区域和名称空间重构

ReSharper 9.0引入了ReSharper平台，该平台是该平台的通用二进制发行版，用于构建ReSharper，dotCover，dotTrace，dotPeek和其他基于.net的JetBrains工具。区域是一种将组件划分为功能集的机制，这些功能集可以启用或禁用，也可以由一种产品（而不是另一种产品）要求。这用于停止指示包装问题的使用要求。也就是说，如果产品A使用了特定的语言功能，但产品B没有使用该功能，则仍可以在针对该语言的单个程序集中实现该功能，而不必将程序集任意拆分为两个，每个装运程序集一个。

现在，组件必须声明它们需要激活的区域。如果任何必需的区域都不可用（由于产品支持或配置），该组件也将不可用。例如，如果某个插件发布了实现“快速修复”的组件，则它必须声明它需要“快速修复”。如果产品不支持快速修复（例如dotPeek或dotTrace），则不会实例化该组件。再加上单一的二进制发行版，意味着插件现已应用于所有产品，而不仅仅是ReSharper。如果相关功能可用，则插件将在该产品中加载。

区域基于名称空间-区域标记器类声明该名称空间及其以下所有组件的必需区域。因此，ReSharper中的许多名称空间都已重命名，以更好地反映命令指定的分层分类区域。

现有的插件作者将必须更新其插件才能使用区域。首先，许多错误将只是名称空间重命名。使用ReSharper自动从其新名称空间中导入类型应该可以解决许多错误。其次，插件作者应添加区域标记类，以描述插件功能所需的区域。ReSharper提供了内部分析来帮助您解决此问题。

有关更多详细信息，请参见区域部分 。

扩展包

扩展现在是ReSharper平台的扩展，而不是ReSharper的扩展。这需要对扩展的打包和部署方式进行一些更改。它们仍然是NuGet包，但是内容的布局已更改。不应将扩展名放在名为“ ReSharper”的根文件夹中，而是将插件，外部注释或设置文件放在名为“ DotFiles”的文件夹中。安装后，扩展程序现在直接安装到产品文件夹中，该文件夹现在位于%LOCALAPPDATA% ， 而不是C:\Program Files 。

有关更多详细信息，请参见“ 部署 ” 部分 。

删除插件元数据属性

的[PluginTitle] ， [PluginVendor]和[PluginDescription]不推荐使用和删除属性。“插件选项”页面也已被删除，这意味着该信息的唯一显示位置是在扩展管理器中，该管理器使用了扩展的程序包元数据中的类似信息。该属性应删除。

ReSharper平台已更改了与扩展程序和插件一起使用的方式。以前，插件具有ReSharper的特殊待遇-插件通过组件模型分别加载到产品组件中，并在“插件选项”页面中列出。ReSharper平台现在将扩展与产品组合视为相同，或者，将产品与扩展视为相同。ReSharper，dotCover和dotPeek等都是ReSharper平台的扩展。在安装过程中，扩展（第一方和第三方）是静态注册的，并在运行时组成以创建单个应用程序。扩展管理器检索扩展并调用安装程序以将新扩展组合到应用程序中。

“插件选项”页面和扩展管理器的先前版本都允许禁用插件或扩展。此功能已删除，有利于在区域级别禁用功能，从而提供了对可以禁用功能的更丰富控制-整个产品，或扩展实现的多个功能之一，例如第一方扩展（例如dotTrace或第三方扩展程序，例如插件。

主要区别在于这是一种选择加入机制。为了允许禁用产品或功能，插件必须将区域广告为产品或功能。区域指南的“ 功能和产品”部分对此进行了说明。

动作

此发行版已重写了Action子系统。最明显的变化是：

  • IActionHandler将被替换IExecutableAction 。
  • 每个动作处理程序都需要一个唯一的ID。
  • 批量操作已被重写。

Visual Studio整合

迁移到共享的ReSharper平台意味着现在可以将产品视为ReSharper平台的扩展。这改变了Visual Studio集成的处理方式。以前的版本将为每个产品创建单独的Visual Studio扩展，例如ReSharper，dotCover和dotTrace将作为单独的扩展单独安装到Visual Studio中。

从ReSharper 9开始，只有一个Visual Studio扩展-ReSharper Platform。产品已安装到ReSharper平台中，该平台将在Visual Studio中更新其注册，以确保所有新程序集都已在Visual Studio中注册。

此过程也适用于扩展，这些扩展的处理方式与产品完全相同-将扩展程序集安装到ReSharper平台后，将在Visual Studio中静态注册。这有两个含义：

  1. ReSharper扩展可以导入和导出MEF组件。
  2. 不再支持扩展“实时加载”，并且在安装扩展时必须始终重新启动Visual Studio。
  3. 扩展程序执行的操作将记住快捷方式分配和自定义。这修复了RSRP-246461和RSRP-337038 。

强烈建议尽可能使用ReSharper抽象而不是MEF接口，因为这允许在Visual Studio外部使用扩展。

后台组件实例化

从ReSharper 9.0开始，组件模型将不再在应用程序启动期间立即创建组件，而是将创建过程推迟到应用程序变得空闲为止。这对应用程序启动性能具有非常积极的影响。

更具体地说，任何标记有属性的类，例如[ShellComponent]和[SolutionComponent]由于组成了零件容器（例如，应用程序启动或解决方案加载），因此不再立即创建，而是排队到当前线程的Dispatcher ， 在Background优先。与以前的版本一样，仍在主线程上创建组件，并且仍然首先创建所有依赖项。

此更改的一个含义是，现在可以在创建容器中的所有组件之前启动并运行应用程序。以前的版本会在允许应用程序继续运行之前创建整个容器。在某些情况下，应更及时地创建组件。例如，需要先创建并连接Visual Studio集成，然后才能继续应用程序。在这种情况下，组件可能需要“即时”激活，在这种情况下，将立即创建组件（以及满足构造函数依赖关系的所有组件），而不是按计划进行创建：

[ShellComponent(Requirement = InstantiationRequirement.Instant)] public class ImmediateComponent { }

有关更多详细信息，请参见“ 组件模型”部分 。

请注意，这仅在ReSharper平台托管在Visual Studio中时适用。独立产品始终会立即实例化组件。

控制流接口重命名

控制流图界面已重命名为正确的拼写：

  • IControlFlowGraf ->IControlFlowGraph
  • IControlFlowRib ->IControlFlowEdge

此外，已将语言特定的接口重命名为：

  • ICSharpControlFlowGraf -> ICSharpControlFlowGraph和ICSharpControlFlowRib ->ICSharpControlFlowEdge
  • IVBControlFlowGraf -> IVBControlFlowGraph和IVBControlFlowRib ->IVBControlFlowEdge
  • IJsControlFlowGraf -> IJsControlFlowGraph和IJavaScriptControlFlowRib ->IJsControlFlowEdge

的IControlFlowBuilder和ControlFlowBuilder提供与语言无关的方法来创建控制流程图。

搜索选项

现在可以搜索选项页面。要在自己的选项页中实施，请实施ISearchablePage ，或从SimpleOptionsPage 。只要页面设置了附加属性，突出显示的结果就会自动为XAML页面实现SearchablePageBehavior.SearchFilter="True"在页面的根控件上，同时还设置了DataContext根控制到执行ISearchablePage 。

单元测试提供者更改

单元测试元素ID更改

单元测试元素的ID不再是简单的字符串ID。此项更改是为了支持同一项目中的多个提供程序（例如，混合使用nunit和xunit测试），并且还允许在同一解决方案中使用具有相同完全限定类或方法名称的测试。这对于共享项目尤其重要。

现在ID不再是简单的字符串，而是UnitTestElementId实例，这是一种值类型（如果字段全部相等，则可以将多个实例视为相等）。它由三个值组成-测试提供程序（例如nunit，xunit）， PersistentProjectId ，通常是.csproj文件，以及元素的旧字符串ID。元素的字符串ID可以是类或方法的完全限定名称-将所有三个值都考虑在内可以实现ID的唯一性。

永久项目ID可以通过调用来检索IProject.GetPersistentId() 。

文件和元数据浏览器

的IUnitTestMetadataExplorer ， IUnitTestFileExplorer和ExploreExternal和ExploreSolution上的方法IUnitTestProvider接口已合并为一个接口， IUnitTestElementSource 。这应该被实现为[SolutionComponent] 。它直接替代了上面提到的接口和方法，从而允许从PSI语法树或程序集IL元数据中发现单元测试。的IUnitTestElementSource.ExploreSolution该方法允许通过其他非规定的方式发现测试元素，可能是通过检查非项目文件的解决方案，或者寻找没有PSI语法树的项目文件。

该接口的实现可以遵循以下方面的现有实现： IUnitTestMetadataExplorer要么IUnitTestFileExplorer ，并进行细微的改动。的UnitTestElementConsumer代表已被更灵活的替代IUnitTestElementsObserver和MetadataElementsSource类提供了用于加载程序集IL元数据和调用ExploreAssembly类方法。
