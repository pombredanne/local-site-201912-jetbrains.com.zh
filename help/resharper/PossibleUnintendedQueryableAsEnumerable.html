代码检查：IQueryable可能无意用作IEnumerable

您可以配置此检查：

  • 禁用它以忽略所有相关问题
  • 在代码中的任何位置都禁止显示它以忽略特定问题
  • 更改其严重性级别，以使检测到的问题更少或更明显

实现IQueryable接口的对象应由LINQ提供程序（通常是远程数据库）及其特定接口实现进行处理。

但是，由于IQueryable继承自IEnumerable ，在某些情况下IQueryable对象被传递并作为IEnumerable对象，意味着所有处理都在本地计算机上完成。这种隐式转换在测试环境中不是问题，但可能会对生产产生巨大的性能影响。

假设您得到了IQueryable数据库中的对象：

可查询资源= myDb。GetResources（）;

...并且在您的代码库中的某个地方，有一个方便的扩展方法可以过滤掉空项目：

公共静态IEnumerable WhereNotNull （此IEnumerable 项）{返回项。其中（x => x！= null）； }

现在，让我们看看如何使用可查询对象：

//在这种情况下，“ Where”将在数据库服务器中处理，我们只得到结果：var notNullResources1 = resources。其中（x => x！= null）； //在第二种情况下，我们将首先从数据库中获取所有Resource对象，然后在本地对其进行处理。//同样，'notNullResources2'将是'IEnumerable'类型，//因此，此对象的所有其他处理也将在本地完成。var notNullResources2 =资源。WhereNotNull（）; // IQueryable可能无意中用作IEnumerable

ReSharper将检测到这种转换，并提出修正使其明确：

var notNullResources2 =资源。AsEnumerable（）。WhereNotNull（）;
