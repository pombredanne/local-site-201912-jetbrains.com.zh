VCS附加组件

首次从版本控制将项目导入MPS或为现有项目添加VCS映射时，MPS建议安装所谓的VCS附加组件（以后也可以从主菜单中安装它们： Version Control | Install MPS VCS附加组件 ）。

VCS附加组件是Subversion和Git的特殊挂钩或合并驱动程序，它们覆盖了特定文件类型的本机合并机制。它们确定模型文件（ * .mps ）和生成的模型缓存（ 依靠 ， 生成的文件和trace.info文件）的合并过程。每次执行与VCS相关的操作（例如merge或rebase branch ）时，都会调用这些挂钩。

对于模型 ，他们阅读其XML内容，并尝试以“模型”术语高级地合并更改，而不是简单地合并可能导致无效标记的.xml文件行。有时，无法自动合并模型，然后它们处于冲突状态，可以手动合并 。例如，在应用了所有无冲突的更改之后，当模型最终以一个具有相同id的节点结束时， id冲突可能会发生。在这种情况下，不会执行自动合并，因为这可能导致节点引用出现问题，从而导致难以识别。您需要检查合并结果并手动解决此类冲突。

对于模型缓存 ，合并驱动程序的工作方式有所不同。合并后，将清除生成器依赖项（ 生成的文件 ）和调试器跟踪缓存（ trace.info文件 ），因此您将需要重新生成相应的模型。使用简单的联合算法合并用于编译的Java依赖项（ 依赖项文件 ），该算法使合并后的编译成为可能。

有几种类型的VCS加载项。

  • Git全局autocrlf设置：强制Git在远程存储库中存储具有标准Unix行尾（LF）的文本文件，而工作副本中的文本文件使用与本地系统有关的行尾。如果您的队友使用具有不同行尾的不同操作系统（Windows和Unix），这将很方便。

  • Git全局合并驱动程序设置：在全局Git设置中注册MPS模型的合并驱动程序，以便可以在.gitattributes文件中引用它们。它仅将合并驱动程序名称及其路径映射到实际的合并驱动程序命令。

  • 存储库的Git文件属性：为特定文件类型（ * .mps ， trace.info 等 ）启用MPS合并驱动程序。它创建一个列出所有现有文件类型的文件，以便合并驱动程序可以识别它们。建议将此文件保留在版本控制下，因为配置将在您的团队成员之间共享。
