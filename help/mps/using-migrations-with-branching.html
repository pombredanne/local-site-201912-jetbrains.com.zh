在分支中使用迁移

为了很好地支持语言演变，迁移会自动升级客户端代码，以使用一种或多种语言的最新版本。迁移文档描述了在MPS中创建和运行迁移的过程。简而言之，为了更改模型以适应语言更改，语言作者可以提供迁移。MPS自动跟踪项目中使用的语言版本，并要求最终用户应用所需的迁移。

尽管使用单个开发分支几乎没有什么惊喜，但典型的团队设置涉及多个分支，分别用于功能和错误修复。拥有多个分支机构的团队将不可避免地面临这样的情况，例如在不同分支机构中使用不同版本的语言，这使合并过程更具挑战性。

无痛合并的主要规则

为了平稳地合并两个分支，在这些分支中使用的所有语言应具有相同的版本。下面介绍了一个简单的工作流程，使您可以轻松地在两个分支中同步语言版本。

在任何合并的分支中均未创建任何迁移

要同步语言版本，只需更新过时的分支中的语言并运行所需的迁移即可。现在，两个分支都具有使用的语言的相同版本，并且可以轻松合并。

已创建一些迁移

所有考虑过的案例背后都有相同的想法： 切勿合并使用语言版本不同的分支 。因此，我们需要在合并之前将所有迁移传播到两个分支中。

1。其中一个分支机构进行了一次迁移

编写迁移时，应该做的唯一特别的事情是，应该有一个包含迁移本身的提交（将其命名为C），但是不应在同一提交中迁移任何代码 。迁移的文件可以稍后提交，例如在以下提交中。
这里的主要思想是使其他分支机构不必合并使用的语言的不同版本即可进行新的迁移。

现在，您的历史记录如下：

| --- C --- [分支1]

------ |

| ------------ [分支2]

合并分支b1和b2

  1. 将b2与提交C合并。现在，您在b2中有了新的迁移。

  2. 在b2上运行迁移

  3. 与b1的HEAD合并

很简单，不是吗？

每当您添加新的语言迁移时，创建一个单独的提交都是一个好习惯。这样一来，其他分支机构就可以安全地进行迁移，而不必理会迁移对分支机构中的代码造成的升级。

2。一个分支中有两个或多个迁移

原理保持不变，只是重复两次或更多次

| ----- C1 — C2 – [分支1]

------ |

| ------------ [分支2]

  1. 将b2与C1合并

  2. 在b2上运行C1的迁移

  3. 与C2合并

  4. 在b2上运行迁移

  5. 与b1合并

3。在两个分支中都创建了迁移

通常，我们建议您避免这种情况，因为这种情况很难解决。通过仅在一个商定的分支（例如，主服务器）中创建所有迁移，可以最好地避免这种情况。

但是，如果遇到这种情况，可以通过以下方式优雅地解决它：

| --- C1 ----- [分支1]

------ |

| --- C2 ----- [分支2]

  1. 合并b2和C1，b1和C2

  2. 在b2上运行C1的迁移，在b1上运行C2的迁移

  3. 合并分支
