升级上游集群

请按照本指南将现有的Upsource群集安装升级到较新的版本。升级过程根据版本增量的不同而有很大差异（即2018.2.1至2018.2.2与2018.1。X至2018.2。X），如下所述。

小升级

升级到仅具有内部版本号不同的版本时，例如从2018.2.0升级到2018.2.2时，请使用此说明。

  1. 停止上游集群并删除其容器：

    ./cluster.sh -H tcp：// ： 停止./cluster.sh -H tcp：// ： rm

  2. 下载 cluster-config- .zip

    重要！ 如果您曾经更改以下文件：

    • cluster.sh
    • docker-compose.yml
    • docker-compose-cluster-init.yml
    • docker-compose-cluster-upgrade.yml

    例如，添加了新的分析器属性），您需要：

    • 重命名旧的更改文件（例如， cluster.sh.bak

    • 将新的cluster.sh和docker-compose.yml文件保存在同一目录中。

    • 将更改和添加的行从旧文件复制到新文件。

  3. 检查cluster.sh中是否设置了正确的UPSOURCE_VERSION

  4. 启动上游集群：

    ./cluster.sh -H tcp：// ： 向上

重大升级

在以下情况下使用此说明：

  • 升级到新的主要版本，例如从2018.1.111到2018.2.111

  • 从ZIP或Docker安装迁移。

  1. 创建现有安装的备份 。

  2. 停止上游集群：

    ./cluster.sh -H tcp：// ： 停止./cluster.sh -H tcp：// ： rm

  3. 在运行Cassandra数据库的主机上，从Cassandra卷映射到的文件夹中删除所有数据。默认目录为/ opt / upsource / cassandra 。实际目录在docker-compose.yml中定义

    大型升级期间无法迁移索引数据。所有用户生成的数据将从备份中还原。

  4. 将特定日期的备份文件夹（假设包含备份的文件夹为/ upsource / backup / 2017 Oct 11 12-18-26 ） 复制到节点$ {UPSOURCE_CLUSTER_INIT_NODE}

  5. 将该备份目录路径添加到文件docker-compose-params.env中的属性UPSOURCE_UPGRADE_SOURCE_BACKUP_DIRECTORY中。

  6. 提供对该备份文件夹的访问权限：

    chown -R 13001：13001“ / upsource / backup / 2017 Oct 11 12-18-26”

  7. 下载 cluster-config- .zip

    重要！ 如果您曾经更改以下文件：

    • cluster.sh
    • docker-compose.yml
    • docker-compose-cluster-init.yml
    • docker-compose-cluster-upgrade.yml

    例如，添加了新的分析器属性），您需要：

    • 重命名旧的更改文件（例如， cluster.sh.bak

    • 将新的cluster.sh和docker-compose.yml文件保存在同一目录中。

    • 将更改和添加的行从旧文件复制到新文件。

  8. 检查cluster.sh中是否设置了正确的UPSOURCE_VERSION

  9. 将以下行添加到upsource.env ：

    UPSOURCE_MONITORING_RECORD_HOURS = 1

  10. 运行以下命令：

    ./cluster.sh upgrade-cluster -H tcp：// ：

    该命令：

    • 停止上游群集服务（如果有任何运行中的服务）。

    • 启动Cassandra数据库并将备份应用到该数据库。

    • 停止卡桑德拉。

  11. 检查命令是否已成功运行。执行日志位于$ {UPSOURCE_CLUSTER_INIT_NODE}节点的目录/ var / log / upsource / cluster-init中。

  12. 启动上游集群：

    ./cluster.sh -H tcp：// ： 向上
