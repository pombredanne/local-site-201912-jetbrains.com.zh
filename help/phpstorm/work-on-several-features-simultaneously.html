PhpStorm 2019.3帮助

同时处理多个功能

有时，您需要在未完成的事情之间切换不同的任务，然后再返回到它们。PhpStorm为您提供了几种方法，可方便地使用多种不同功能，而又不会丢失您的工作：

  • 您可以隐藏或搁置未决的更改。

    存放更改与搁置非常相似。唯一的区别在于补丁的生成和应用方式。污渍是由Git生成的，可以在PhpStorm内部或外部应用。具有搁置更改的补丁程序是由PhpStorm生成的，也可以通过IDE应用。此外，存储涉及所有未提交的更改，而将更改放入架子时，您可以选择一些本地更改而不是将其全部搁置。

  • 您可以将与不同任务或功能相关的更改保留在不同的更改列表中 。

  • 您可以创建分支以使用不同的不相关功能 。

搁置更改

搁置架暂时存储您尚未提交的未决更改。例如，如果您需要切换到另一个任务，并且想将更改留在一边以便以后进行处理，则这很有用。

使用PhpStorm，您可以搁置单独的文件和整个变更列表 。

搁置后，可以根据需要多次应用更改。

搁置更改

  1. 打开 版本控制窗口Alt+9并切换到“ 本地更改”标签。

  2. 右键单击要放在架子上的文件或更改列表，然后从上下文菜单中选择“ 搁置更改 ”。

  3. 在“ 搁置更改”对话框中，查看已修改文件的列表。

  4. 在“ 提交消息”字段中，输入要创建的架子的名称，然后单击“ 搁置更改”按钮。

您也可以静默搁置更改，而无需显示“ 搁置更改”对话框。为此，请选择要搁置的文件或更改列表，然后单击“ 静默搁置”图标在工具栏上，或按Ctrl+Shift+H 。包含要搁置的更改的更改列表的名称将用作架子名称。

为避免结成多个同名的架子（例如Default ），您可以将文件或更改列表从“ 本地更改”选项卡拖到“ 架子”选项卡，等待一秒钟直到它被激活，然后编辑新的架子释放鼠标按钮时即时命名。

搁置更改

搁置正在将已推迟的更改从磁盘架移至待处理的更改列表。搁置的更改可以从视图中过滤掉或从架子上删除。

  1. 在“ 货架”标签中 版本控制窗口Alt+9 ，选择您要搁置的更改列表或文件。

  2. 按Ctrl+Shift+U或从所选内容的上下文菜单中选择“ 搁置 ”。

  3. 在打开的“取消搁置的更改”对话框中，在“ 名称”字段中指定要将未搁置的更改还原到的更改列表。您可以从列表中选择一个现有的更改列表，或为要创建的包含未搁置更改的新更改列表键入名称。您可以在“ 注释”字段（可选）中输入新变更列表的描述。

    如果要使新变更列表处于活动状态，请选择设置活动选项 。否则，当前活动的更改列表将保持活动状态。

  4. 如果你想PhpStorm保留在其停用了新的修改相关联的任务的上下文和恢复的背景下，当变更列表变得活跃，选择曲目方面选择（见任务和环境的详细信息）。

  5. 如果要删除将要搁置的更改，请选择“ 从架子中删除成功应用的文件”选项。未搁置的文件将从此架子中删除，并添加到另一个变更列表中，并标记为已应用。在通过单击将其明确删除之前，它们不会被完全删除。 在工具栏上，或从上下文菜单中选择“ 清理已搁置的书架 ”。

    如果您意外删除了未搁置的文件，则可以从“ 最近删除”节点中查看和还原它们。

  6. 单击确定 。如果修补程序版本和当前版本之间发生冲突，请按照解决冲突中所述解决它们。

您可以静默取消搁置更改，而无需显示“取消搁置更改”对话框。为此，请选择要取消搁置的文件或更改列表，然后单击“ 静默取消搁置”图标在工具栏上，或按Ctrl+Alt+U 。未搁置的文件将被移至活动的挂起变更列表。

您也可以将文件或更改列表从“ 货架”选项卡拖动到“ 本地更改”选项卡以静默取消搁置。如果您按住它拖动Ctrl键，它将被复制到“ 本地更改”标签，而不是从机架中删除。

恢复搁置的更改

PhpStorm使您可以在必要时重新应用未搁置的更改。所有未搁置的更改都可以重复使用，直到通过单击工具栏上的图标，或从上下文菜单中选择“ 清理已搁置的已搁置 ”。

  1. 确保节目已被搁置 工具栏选项已启用。

  2. 选择要还原的文件或文件架。

  3. 从选择的上下文菜单中，选择“还原”。

应用外部补丁

您可以导入在PhpStorm内部或外部创建的补丁，并将其应用为搁置更改。

  1. 在“ 货架”标签中 版本控制窗口Alt+9 ，从上下文菜单中选择“ 导入补丁 ”。

  2. 在打开的对话框中，选择要应用的补丁文件。选定的色块将在架子上显示为架子。

  3. 选择带有补丁的新添加的架子，然后从所选内容的上下文菜单中选择“ 取消搁置更改 ”。

自动搁置基础修订

将PhpStorm配置为始终搁置受Git版本控制的文件的基本修订版可能会很有用。

  1. 在里面 设置/首选项对话框Ctrl+Alt+S ，选择版本控制|货架 。

  2. 选择“ 在分布式版本控制系统下搁置文件的基本修订”选项。

    如果启用此选项，则文件的基本修订版将保存到文件架中，如果应用架子导致冲突，则该文件将在三路合并期间使用。如果禁用，则PhpStorm将在项目历史记录中查找基本修订，这可能需要一段时间。此外，可能会遗漏有冲突的架子所基于的修订版本（例如，如果由于进行了变基操作而更改了历史记录）。

更改默认的货架位置

默认情况下，架子目录位于您的项目目录下。但是，您可能要更改默认的架子位置。例如，如果您要避免在清理工作副本时意外删除架子，或者要将它们存储在一个单独的存储库中，以允许团队成员共享架子，这将很有用。

  1. 打开设置 首选项对话框Ctrl+Alt+S然后选择版本控制|左侧的货架节点。

  2. 单击更改货架位置按钮，然后在打开的对话框中指定新位置。

  3. 如有必要，请选择将书架移至新位置选项，以将现有书架移至新目录。

观看此视频教程，了解如何从货架中受益，从而能够切换到其他任务而又不会丢失未完成的工作：

藏匿变化

有时可能需要还原工作副本以匹配HEAD提交，但您不想丢失已经完成的工作。如果您了解到上游更改可能与您正在做的事情有关，或者您需要进行一些紧急修复，则可能会发生这种情况。

存放涉及记录HEAD提交与工作目录的当前状态（存放）之间的差异。索引的更改也可以隐藏。

取消隐藏涉及将存储的隐藏项应用于分支。

您可以将存储应用于现有分支或在其基础上创建一个新分支。

存储可以根据需要多次应用到所需的任何分支，只需切换到所需的分支即可 。请记住：

  • 在一系列提交之后应用存储将导致需要解决的冲突。

  • 您不能将存储应用于“脏”工作副本，即具有未提交更改的工作副本。

将更改保存到存储中

  1. 从主菜单中，选择VCS |。 Git |藏匿变化 。

  2. 在打开的对话框中藏匿 ，选择合适的Git的根，并确保正确的分支被检查出来。

  3. 在消息字段中，描述您将要存储的更改。

  4. 要存储本地更改并将索引中暂存的更改带入工作树以进行检查和测试，请选择“ 保留索引”选项。

  5. 单击创建存储 。

申请藏匿处

  1. 从主菜单中，选择VCS |。 Git |取消更改。

  2. 选择要在其中应用隐藏的Git根，并确保签出了正确的分支。

  3. 从列表中选择要应用的存储。

    如果要检查所选存储中受影响的文件，请单击查看 。

  4. 要在应用后删除所选存储，请选择“ 弹出存储”选项。

  5. 要也应用隐藏索引修改，请选择“ 恢复索引”选项。

  6. 如果要基于所选存储区创建新分支，而不是将其应用于当前已检出的分支，请在“ 作为新分支”字段中键入该分支的名称。

要删除存储，请在列表中将其选中，然后单击“ 放置” 。要删除所有隐藏物，请单击清除 。

将更改分组到不同的更改列表中

当您使用多个相关功能时，可能会发现将更改分组到不同的更改列表中很方便。与使用要素分支来完成多个任务相比，这种方法有其优缺点。

优点 ：

  • 您可以轻松地在不同的逻辑更改集之间进行切换，并将它们彼此分开提交。

  • 与将分支用于相同的目的不同，您可以立即进行所有更改，而不必在分支之间进行切换，如果您的项目很大，则可能需要一段时间。

  • 测试不同功能如何协同工作很方便。

  • 您可以在构建服务器上远程运行变更列表。

缺点 ：

  • 尽管与分支机构相比，使用变更表似乎是一种更轻量级的选择，但这并不安全，因为在提交并推送变更之前，您无法备份任何变更。如果您的本地工作副本发生了某些事情，您所有的更改都将丢失，因为它们不是Git项目历史记录的一部分。

  • 无法对功能进行原子测试。

  • 无法在同一功能上进行协作。另外，除非您通过电子邮件发送带有更改的补丁，否则可能无法从其他计算机中进行贡献，这可能不太方便。

所有变更列表均显示在 的“本地更改”标签 版本控制窗口Alt+9 。除非您创建了另一个文件并将其激活，否则所有修改过的文件都会自动放置在活动更改列表中，该列表即为默认更改列表。

要创建新的更改列表，请单击在工具栏上。

要使非默认变更列表处于活动状态，请右键单击它，然后从上下文菜单中选择“ 设置活动变更列表”。

在变更列表之间移动变更

  1. 选择要移动的文件，然后从上下文菜单中选择“ 移至另一个变更列表 ”。

  2. 如果要使用即将放弃活动更改列表的更改来制作更改列表，请选择“ 设置活动选项”。

  3. 如果您希望PhpStorm记住您的上下文并在此更改列表变为活动状态时在编辑器中重新加载当前打开的文件，请选择“ 跟踪上下文”选项。

您也可以在变更列表之间拖动文件。

使用功能分支

Git中的分支代表独立的开发线，因此，如果您要使用单独的功能，则在准备共享工作结果并将其集成到其中之前，要完成并测试master ，在功能分支中执行此操作是最佳解决方案。这样，您可以确保不稳定的代码不会提交给项目的主要代码库，并且在必要时可以轻松切换到其他任务。

优点 ：

  • 与使用更改列表将更改分组相比，使用功能分支是安全的。提交对Git的更改后，它们将成为Git项目历史的一部分，因此即使您破坏了工作树，也可以始终通过Git reflog恢复提交。推送更改后，将备份它们。

  • 您可以开发并行的无关功能并进行原子测试。

  • 在分支中完成开发后，可以重新排序或压缩commit ，以便您的历史记录是线性且整洁的。

  • 您可以轻松地在功能上进行协作，也可以从其他机器上进行开发。

缺点 ：

  • 在大型项目上切换分支机构可能需要一些时间。

  • 一起测试相关功能不是很方便。

  • 您必须学习使用功能分支并将更改集成到主代码库中的工作流程 。

使用功能分支并将更改集成到主要代码库中有两种主要方法：

  • 合并选项

  • 变基选项

使用合并功能集成功能分支中的更改

merge选项的主要优点是完全可追溯性，因为合并到主代码库中的提交保留了其原始哈希值和作者，并且可以将一个功能中的所有提交归为一组。

此工作流程非常适合对主要代码库进行更改涉及牵项请求或层次批准程序的项目，因为现有分支不会以任何方式更改。

这种方法的主要缺点是每次需要合并更改时都会创建无关的合并提交，这会严重污染项目历史并使其难以阅读。

  1. 为您的单独开发线创建一个分支 。

  2. 在开发过程中提交更改。

  3. 将分支推送到远程存储库。应该完成此操作以进行备份，以便您可以在其他计算机上进行协作或工作。

  4. 当您需要执行与功能无关的工作时，请切换到其他分支。

  5. 审查并测试您的功能，并进行必要的修复。

  6. 准备好将工作结果集成到主分支中时（例如master ）， 请执行下列操作：

    • 将功能分支合并到主代码库中。

    • 删除功能分支 。

    • 推。

使用rebase集成功能分支中的更改

此选项的主要好处是您可以获得干净的项目历史记录，其他人也容易阅读和理解。您的日志不包含由merge操作，您将获得易于浏览和搜索的线性历史记录。

在决定采用此工作流程时，请记住， rebase重写项目历史记录，因为它会为原始功能分支中的每个提交创建新的提交，因此它们将具有不同的哈希值，这会妨碍可追溯性。

  1. 为您的单独开发线创建一个分支 。

  2. 在开发过程中经常提交更改。

  3. 将分支推送到远程存储库。应该完成此操作以进行备份，以便您可以在其他计算机上进行协作或工作。

  4. 衍合你的特性分支到master时。仅当功能分支很长时，才有意义。这对以下操作很有用：

    • 确保您的功能分支和master不要分开太远。

    • 最终将更改集成到主代码库时，请避免解决众多冲突。当您定期进行基准调整时，您可以迭代地解决冲突，并且不会因长时间差异而陷入困境。

    • 加快检查分支的速度，因为分支之间的切换一旦充分分散，就会变慢。

    变基涉及以下步骤：

    • 从远程获取更改，或将更改拉入master科。

    • 衍合你的分支到master 。

    • 力推结果rebase操作功能分支。

  5. 切换到master当您需要执行与功能无关的工作时。当您回到功能分支时，请执行Checkout并将Rebase设置为Current 。

  6. 审查并测试您的功能，并进行必要的修复。

  7. 功能完成后，请执行Interactive Rebase 。这使您可以重新排序和压缩提交，以使您的功能分支历史记录看起来更干净。

  8. 准备好将工作结果集成到主分支中时（例如master ）， 请执行下列操作：

    • 结帐 master科。

    • 与您的分支合并 master 。以来master还没有发散，Git只会将指针向前移动到功能分支的最新提交，而不是创建新的合并提交（这称为快进合并）。

    • 删除功能分支 。

    • 推 。
