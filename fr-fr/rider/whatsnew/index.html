Améliorations du débogueur

Prise en charge de Xamarin

Cette version ajoute quelques fonctionnalités utiles que vous apprécierez dans votre développement iOS en Xamarin.

Améliorations des performances

Après la sortie de Rider 2018.3, nous avons reçu de nombreux commentaires sur les performances, et nous nous sommes donc concentrés sur ce point :

  • Depuis que nous avons déplacé la prise en charge d'Android en Xamarin vers une extension indépendante et séparée, le délai d'affichage de l'écran avec la barre de progression après avoir cliqué sur l'icône de Rider a été considérablement réduit. À signaler également, l'outil d'installation est désormais plus léger de 150 Mo.
  • Dans l'une des versions précédentes, le lancement de NGen (Native Image Generator) pour les binaires Rider ne fonctionnait pas. En conséquence, l'ouverture d'une solution dans Rider prenait 8 secondes de plus sous Windows, car le processus de compilation JIT devait s'exécuter au moment du lancement. Maintenant que NGen refonctionne, nous avons regagné 8 secondes à chaque ouverture de solution.
  • Nous avons opéré quelques optimisations du démarrage à chaud : nous avons arrêté d'utiliser le moteur ReSharper dans la prise en charge JavaScript/TypeScript, et rendu le chargement du modèle de projet asynchrone. Suite à ces modifications, les démarrages à chaud sont maintenant considérablement plus rapides (jusqu'à 2 fois).
  • Nous avons implémenté une charge précoce en arrière-plan pour accélérer le chargement du service de langage de ReSharper de 3 secondes. Vous remarquez la différence si le paramètre “Reopen last solution on startup” est activé, ou si vous êtes incroyablement rapide, au point de pouvoir cliquer sur une solution sur l'écran d'accueil en moins d'une seconde.

Publishing to IIS or FTP

Rider introduces two new run configurations: Publish to IIS and Publish to custom server. Both serve the purpose of deploying our ASP.NET Core web application to a local or remote server, but they use different means to do this:

  • Publish to IIS uses MSBuild and MSDeploy to package and publish our application. This run configuration type can deploy our ASP.NET Core web application to local and remote IIS and to Kudu (Azure Web Apps), as well as create an MSDeploy package. Publish profiles come as .pubxml files.

Right now, only publishing of ASP.NET Core Web projects is supported. Additionally, using the Publish to IIS run configuration type to deploy to IIS will only work on Windows.
